# Runtime launch boundary

This document defines the proposed responsibility boundary for runtime launch
behavior before moving `run_server` or runtime dispatch code.

It is a design plan only. Do not move functions, change launch commands, change
signal handling, change shutdown behavior, change lifecycle hook timing, change
install order, or change install-only behavior as part of this document.

## Proposed boundary

Recommended file: `scripts/lib/runtime_launch.sh`

Status: started. `run_server` has moved mechanically to
`scripts/lib/runtime_launch.sh`. Runtime-specific dispatch remains in
`entrypoint.sh`, and shutdown/RCON/signal handling, lifecycle hook orchestration,
command-line mode selection, and install-only orchestration have not moved.

Use `scripts/lib/runtime_launch.sh` because the boundary is about launching the
selected runtime after install/bootstrap has completed. Avoid
`scripts/lib/server_runtime.sh` because it sounds broad enough to include type
resolution, install markers, and runtime state. Avoid `scripts/lib/run_server.sh`
because the file may eventually own runtime-specific launch dispatch as well as
the single `run_server` process wrapper.

The future library should answer:

- How is the selected runtime launched?
- Which command is used for each supported runtime type?
- How are Java launch arguments applied at server start?
- How are Forge and NeoForge launched through `run.sh`?
- How is Velocity launched through `velocity.jar`?
- How is the server process executed and waited on?

`entrypoint.sh` should continue to answer:

- When preflight, type resolution, install/bootstrap, install-only exit, and
  runtime launch happen.
- When lifecycle phases run relative to install and runtime launch.
- How command-line modes such as `run`, `install-only`, `rcon`, `rcon-say`, and
  `rcon-stop` are selected.
- How signal handling and shutdown orchestration are wired unless a later
  dedicated boundary moves them intentionally.

## Current state

Runtime launch orchestration is currently implemented in `entrypoint.sh`.

Current launch-related functions and variables:

- `run_server`
  - Implemented in `scripts/lib/runtime_launch.sh`.
  - Starts the provided command in the background.
  - Stores the process id in `SERVER_PID`.
  - Calls `cleanup_rcon_lock_on_boot` before launching.
  - Waits for `READY_DELAY` seconds, failing early if the process exits.
  - Creates `${DATA_DIR}/.ready` while the process is still running.
  - Waits for the process and removes `${DATA_DIR}/.ready` on exit.
- `runtime`
  - Logs `Starting runtime (TYPE=${TYPE})`.
  - Runs `run_phase_hooks "pre-runtime"` before dispatch.
  - Dispatches by `TYPE`.
  - Calls `run_server` with the selected launch command.
- `JVM_ARGS_FILE`
  - Defaults to `${DATA_DIR}/jvm.args`.
  - Is generated by `install_jvm_args`.
  - Is consumed by Java-based runtime launch commands through
    `java @"${JVM_ARGS_FILE}"`.
- `SERVER_PID`
  - Tracks the background server process started by `run_server`.
  - Is read by shutdown waiting and signal handling.
- `MC_PID`
  - Is currently declared as a reserved global for PID-oriented lifecycle
    handling.
  - Current launch and shutdown behavior uses `SERVER_PID`.
- `INSTALL_ONLY`
  - Is handled in `main` after `install`.
  - When true, logs `INSTALL_ONLY=true, skipping runtime launch and exiting`
    and exits before `runtime`.

Current runtime dispatch:

- `fabric`
  - Launches `java @"${JVM_ARGS_FILE}" -jar
    "${DATA_DIR}/fabric-server-launch.jar" nogui`.
- `quilt`, `paper`, `purpur`, `spigot`, `mohist`, `taiyitist`, `youer`, and
  `vanilla`
  - Launch `java @"${JVM_ARGS_FILE}" -jar "${DATA_DIR}/server.jar" nogui`.
- `forge` and `neoforge`
  - Change directory to `${DATA_DIR}`.
  - Validate `./run.sh` is executable.
  - Run `chmod +x ./run.sh`.
  - Launch `./run.sh nogui`.
- `velocity`
  - Validates `${DATA_DIR}/velocity.jar` exists.
  - Changes directory to `${DATA_DIR}`.
  - Launches `java @"${JVM_ARGS_FILE}" -jar "${DATA_DIR}/velocity.jar"`.

Current adjacent behavior:

- Signal handling uses `trap 'graceful_shutdown' TERM INT QUIT`.
- `graceful_shutdown` skips `rcon_stop` for `TYPE=velocity`, otherwise calls
  `rcon_stop_once` and falls back to sending `TERM` to `SERVER_PID`.
- `wait_for_server_exit` polls `SERVER_PID`.
- `rcon_stop_once` uses `RCON_STOP_LOCK`, `RCON_STOP_IN_PROGRESS`, and
  `RCON_STOP_RESULT` to avoid duplicate RCON stop behavior.
- `rcon_exec`, `rcon_say`, `rcon_stop`, and `rcon_tellraw_all` are adjacent to
  launch because shutdown can invoke them.
- Command modes `rcon`, `rcon-say`, and `rcon-stop` are selected before `main`.
- A Velocity foreground launch footer exists after the sourced guard. It checks
  `TYPE=velocity`, requires `${DATA_DIR}/velocity.jar`, changes to
  `${DATA_DIR}`, and `exec`s `java -jar "${DATA_DIR}/velocity.jar"` if reached.

## What should move

Future `scripts/lib/runtime_launch.sh` may own:

- `run_server`.
- Runtime-specific launch command dispatch.
- Java command construction for server launch.
- Forge and NeoForge `run.sh` launch behavior.
- Velocity `velocity.jar` launch behavior.
- `JVM_ARGS_FILE` handling if it is only used for launch.
- Launch-time validation directly tied to starting the runtime.
- Server process execution behavior.

## What should not move

Future `scripts/lib/runtime_launch.sh` should not own:

- Server artifact installation.
- `install_server`.
- `TYPE=auto` resolution.
- Runtime marker helpers.
- Velocity TOML generation.
- `server.properties` bootstrap.
- World install or world reset.
- S3/MinIO setup.
- Plugin, mod, config, datapack, or resourcepack installation.
- Command-line mode selection.
- Install-only orchestration.
- RCON shutdown policy unless the current code makes it inseparable.
- Lifecycle hook orchestration unless the current code makes it inseparable.
- Signal trap registration unless a dedicated shutdown boundary moves it.

## Current coupling

The launch boundary is higher risk than prior mechanical moves because launch
code is close to process and shutdown behavior.

Current couplings to preserve during any mechanical move:

- `run_server` depends on `DATA_DIR`, `READY_DELAY`, `SERVER_PID`,
  `cleanup_rcon_lock_on_boot`, `log`, and `die`.
- `runtime` depends on `TYPE`, `DATA_DIR`, `JVM_ARGS_FILE`, `run_phase_hooks`,
  `run_server`, `log`, and `die`.
- Java-based runtime branches depend on `JVM_ARGS_FILE` existing before launch.
- Forge and NeoForge launch from `${DATA_DIR}` through `./run.sh nogui`.
- Velocity launch changes to `${DATA_DIR}` and uses `${DATA_DIR}/velocity.jar`.
- Shutdown behavior depends on `SERVER_PID` being set by `run_server`.
- Signal handling calls `graceful_shutdown`, which may use RCON before sending
  process signals.
- `pre-runtime` lifecycle hooks currently run inside `runtime` before dispatch.
- Install-only behavior remains orchestration-owned in `main`; it exits before
  `runtime` is called.
- The Velocity foreground footer is separate from `runtime` dispatch and should
  not be moved casually.

## Staged migration order

Recommended implementation PRs:

1. Add this docs-only boundary plan.
   - Status: completed.
2. Mechanically move tiny launch-only helpers, if any are identified.
   - Preserve call sites and source order.
   - Do not move RCON, shutdown, signal, or lifecycle behavior in this step.
3. Mechanically move `run_server` into `scripts/lib/runtime_launch.sh` only if it
   can be moved without behavior changes.
   - Preserve readiness file behavior.
   - Preserve `SERVER_PID` assignment.
   - Preserve wait and return-status behavior.
   - Preserve cleanup of the RCON stop lock if it remains coupled.
   - Status: completed for `run_server` only.
4. Keep process orchestration, signal handling, lifecycle hooks, and RCON
   behavior in `entrypoint.sh` unless they are inseparable from `run_server`.
5. After `runtime_launch.sh` stabilizes, consider dedicated behavior-neutral
   boundaries for lifecycle hooks, RCON, and shutdown/signal handling.
6. Do not combine runtime launch moves with cleanup backlog items.

## Risk notes

Specific risks to avoid during extraction:

- Changing launch command arguments.
- Changing `JVM_ARGS_FILE` behavior.
- Changing working directory behavior.
- Changing Forge or NeoForge `run.sh` launch behavior.
- Changing Velocity launch behavior.
- Changing readiness file creation or cleanup.
- Changing process or PID handling.
- Changing signal propagation.
- Changing shutdown behavior.
- Changing RCON stop behavior.
- Changing lifecycle hook timing.
- Changing install-only mode behavior.
- Accidentally mixing install and launch responsibilities.

## Test and smoke guidance

For future implementation PRs that create or populate
`scripts/lib/runtime_launch.sh`, use focused checks:

- Run `bash -n entrypoint.sh scripts/lib/*.sh`.
- Run `shellcheck -x -s bash entrypoint.sh scripts/lib/*.sh` when ShellCheck is
  installed.
- Add or run a source smoke test for `scripts/lib/runtime_launch.sh`.
- Avoid a real server boot unless existing CI already does it safely.
- Keep install-only smoke focused on install behavior.
- Prefer launch command smoke that avoids network calls and long-running server
  processes.
- Use temporary directories or mocked commands when practical.
- Check launch command arguments and working directory behavior exactly.
- Preserve launch command output and arguments unless a dedicated behavior PR
  changes them.
- Do not require S3 credentials.
- Do not trigger destructive world reset behavior.

## Guardrails

- Keep future implementation PRs small and mechanical.
- Do not move `run_server` and broad runtime dispatch together unless the move is
  still mechanically reviewable.
- Do not move shutdown/RCON/signal handling in the same PR as the first launch
  move unless it is truly inseparable.
- Do not change launch semantics while moving ownership.
