# Nightly Triage Memo

## Summary

- Custom `LEVEL_NAME` resets back up `/data/world` but delete `/data/<LEVEL_NAME>`, creating a world-loss risk.
- Mods/config activation replaces complete `/data` directories even when `*_REMOVE_EXTRA=false`.
- Plugin download failures can still remove existing plugins during activation when remove-extra is enabled.
- Image publishing runs independently of CI, allowing untested or failing commits and tags to be published.
- Several committed smoke tests are syntax-checked but never executed.
- AWS CLI migration coverage and documentation remain incomplete or stale.

## High Priority Issue Candidates

### 1. Custom world reset backs up the wrong directory

- Type: bug
- Priority: P1
- Confidence: high
- Impact: A reset using `LEVEL_NAME=custom-world` can delete `/data/custom-world` without backing it up.
- Evidence: `reset_world` derives its deletion target through `minecraft_world_dir`, but `create_world_reset_backup` always runs `tar -C "${data_dir}" world`. The custom-level smoke disables backups and therefore does not cover this combination.
- Why this matters: This violates the repository’s central guarantee that world deletion must not proceed without a valid backup when backups are enabled.
- Suggested approach: Pass the selected world basename to the backup helper and archive exactly that directory.
- Acceptance criteria: A smoke test with `LEVEL_NAME=custom-world` and `RESET_WORLD_BACKUP=true` verifies the archive contains `custom-world/level.dat`, excludes `/data/world`, and deletion occurs only after archive creation succeeds.
- Risk: Medium; this touches destructive reset behavior.
- Suggested labels: bug, world-safety, backup, P1

### 2. Remove-extra false does not preserve local mods or configs

- Type: bug
- Priority: P1
- Confidence: high
- Impact: User-managed files under `/data/mods` or `/data/config` can disappear after startup despite `MODS_REMOVE_EXTRA=false` or `CONFIGS_REMOVE_EXTRA=false`.
- Evidence: `activate_dir` creates a staging directory exclusively from the input source and then replaces the complete destination directory. `activate_mods` and `activate_configs` use this helper regardless of remove-extra settings.
- Why this matters: README states that remove-extra false is the safer default and implies local files are preserved unless the remote prefix is explicitly authoritative.
- Suggested approach: Define separate merge and authoritative activation modes. Preserve destination-only files when remove-extra is false; use atomic replacement only when explicitly enabled and after remote validation.
- Acceptance criteria: Tests prove a destination-only file survives activation with remove-extra false and is removed only with remove-extra true.
- Risk: High; changing activation semantics can affect existing GitOps users who currently rely on implicit replacement.
- Suggested labels: bug, persistence, s3, mods, configs, P1

### 3. Plugin download failures can remove working plugins

- Type: bug
- Priority: P1
- Confidence: high
- Impact: A transient S3 failure can remove an existing plugin from `/data/plugins`.
- Evidence: `install_plugins` defaults to non-strict behavior and returns success after download errors. Although its remove-extra pass is skipped when errors occur, `activate_plugins` subsequently removes destination jars absent from staging whenever `PLUGINS_REMOVE_EXTRA=true`.
- Why this matters: The code logs that remove-extra runs only after a zero-error sync, but the activation phase does not retain that state.
- Suggested approach: Make failed plugin synchronization prevent removal during activation, preferably by publishing a completed staging set only after all required downloads succeed.
- Acceptance criteria: A test with one failed download, one existing destination plugin, non-strict mode, and remove-extra enabled verifies the existing plugin remains.
- Risk: Medium; plugin update and authoritative-prefix behavior must remain predictable.
- Suggested labels: bug, plugins, s3, persistence, P1

### 4. Publish workflow is not gated by CI success

- Type: release
- Priority: P1
- Confidence: high
- Impact: Every main push can publish moving image tags even if lint or runtime smoke fails. Tag pushes publish without the branch-only CI workflows testing that tag.
- Evidence: `publish.yml` triggers directly on main and `v*` pushes and has no dependency on lint or runtime smoke workflows.
- Why this matters: Broken images can replace production-facing tags before failures are noticed.
- Suggested approach: Publish through a successful CI completion event, consolidate build/test/publish into a dependent workflow, or require a verified commit SHA before pushing.
- Acceptance criteria: Failed lint or runtime smoke prevents all image pushes; release tags are tested at the exact tagged commit before publication.
- Risk: Medium; workflow event and permission handling require careful validation.
- Suggested labels: release, ci, docker, reliability, P1

### 2. Custom mods and config input directories are ignored during activation

- Type: bug
- Priority: P2
- Confidence: high
- Impact: Sync can write to a configured staging directory while activation reads stale data from `/mods` or `/config`.
- Evidence: Install functions honor `INPUT_MODS_DIR` and `INPUT_CONFIG_DIR`; activation functions hard-code `/mods` and `/config`.
- Why this matters: The exposed variables do not work end-to-end and could activate unintended files.
- Suggested approach: Resolve each input path once and use it consistently for synchronization and activation.
- Acceptance criteria: Tests verify custom input paths are honored and default paths remain compatible.
- Risk: Medium; custom path support may currently be accidental rather than contractual.
- Suggested labels: bug, configuration, persistence, P2

### 3. Add image-level AWS CLI migration coverage

- Type: test
- Priority: P2
- Confidence: high
- Impact: Packaging or behavioral differences between Debian AWS CLI and GPU AWS CLI v2 may escape helper mocks.
- Evidence: CPU images install distro `awscli`; GPU installs AWS CLI v2. Current image checks largely verify `aws --version`, while S3 helper tests mock the command outside the built image.
- Why this matters: Endpoint handling, legacy credential mapping, and sync argument conversion are core migration compatibility points.
- Suggested approach: Run mocked AWS helper flows inside both CPU and GPU images and assert endpoint, credential, list, copy, sync, and delete argument behavior.
- Acceptance criteria: Both image families exercise the same migration compatibility matrix.
- Risk: Low.
- Suggested labels: test, docker, s3, migration, P2

### 4. Kubernetes examples omit graceful shutdown safeguards

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Users copying the hardcore and GPU manifests receive no RCON configuration, preStop hook, readiness probe, or explicit termination grace period.
- Evidence: `fabric-basic.yaml` contains these controls, while `fabric-hardcore-smp.yaml` and the GPU example do not.
- Why this matters: The examples mount persistent worlds and are presented as runnable Kubernetes configurations.
- Suggested approach: Apply the established shutdown and readiness pattern consistently or clearly label incomplete manifests.
- Acceptance criteria: Persistent-world examples use single-writer rollout, RCON secret, preStop, readiness, and an appropriate grace period.
- Risk: Low.
- Suggested labels: docs, kubernetes, shutdown, P2

### 5. Document a safe backup restore procedure

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Operators can create reset backups but lack a documented, supported restore sequence.
- Evidence: Reset and installation safety are extensively documented, but no dedicated restore workflow exists.
- Why this matters: Recovery instructions are most needed after destructive or failed operations.
- Suggested approach: Document stopping all writers, inspecting archive layout, preserving the current world, restoring the configured `LEVEL_NAME`, ownership checks, and controlled restart using fish-compatible commands.
- Acceptance criteria: Documentation includes archive validation and custom world-name handling without Bash-only heredocs.
- Risk: Low.
- Suggested labels: docs, backup, restore, operations, P2

- Suggested approach: Perform a mechanical cleanup after behavioral tests cover failure and remove-extra cases.

## Suspicious Areas That Need Human Judgment

- CPU images use distribution AWS CLI while the GPU image uses AWS CLI v2. Provider-specific behavior may differ despite matching helper arguments.
- `AWS_EC2_METADATA_DISABLED` defaults to true, which may prevent EC2 instance-profile discovery for users running containers directly on EC2.
- `activate_resourcepacks` exists but is intentionally not called; README describes local synchronization without activation, so this may be retained future functionality.
- World installation accepts exactly one direct `.zip` under a prefix. This is conservative but may conflict with operators expecting versioned archives in one prefix.
- The shutdown sequence can exceed a 120-second Kubernetes grace period when multiple RCON commands exhaust all retries.
- Resourcepack public URL generation validates HTTP/HTTPS shape but does not verify the configured object exists.
