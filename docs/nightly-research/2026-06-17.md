# Nightly Triage Memo

## Summary

- CI has several smoke scripts that are syntax-checked but not executed, including tests for install order, server.properties env mapping, reinstall mismatch behavior, and safe remove-extra defaults.
- Several MinIO `mc` remediation docs are now stale after the AWS CLI replacement and still describe `/usr/local/bin/mc`, `mc-builder`, `configure_mc_alias`, and `mc mirror`.
- The S3/AWS CLI migration has good helper-level mocks, but image-level runtime coverage does not appear to exercise real in-container AWS CLI sync behavior with endpoint/credential mapping.
- `INPUT_MODS_DIR` and `INPUT_CONFIG_DIR` are used during install sync, but activation still reads hard-coded `/mods` and `/config`, making custom staging directories inconsistent.
- Shutdown docs and examples recommend `terminationGracePeriodSeconds: 120`, but default RCON retry/wait settings can plausibly consume more than that in failure or slow-save cases.
- Some examples still set `MAX_MEMORY`, but current runtime code appears to use `JVM_XMS` and `JVM_XMX` only.

### 2. Fix custom input directory activation drift

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Operators setting custom `INPUT_MODS_DIR` or `INPUT_CONFIG_DIR` can sync assets into one directory and activate from another.
- Evidence: `install_mods` writes to `${INPUT_MODS_DIR}`, but `activate_mods` calls `activate_dir "/mods" "${DATA_DIR}/mods"`. `install_configs` writes to `${INPUT_CONFIG_DIR:-/config}`, but `activate_configs` calls `activate_dir "/config" "${DATA_DIR}/config"`.
- Why this matters: Asset install/activation is persistent-state-sensitive. A custom staging dir could be ignored, while stale files under `/mods` or `/config` are activated instead.
- Suggested approach: Make activation use the same resolved input directory as install, with path safety checks matching resourcepacks.
- Acceptance criteria: Smoke tests prove custom `INPUT_MODS_DIR` and `INPUT_CONFIG_DIR` are honored end-to-end and hard-coded defaults still work.
- Risk: Medium, because changing activation paths touches existing mount conventions.
- Suggested labels: bug, s3, persistence, P1

- Risk: Low if mocked; avoid real object storage.
- Suggested labels: test, s3, docker, migration, P1

### 2. Revisit Kubernetes shutdown grace recommendations against default timeout math

- Type: docs
- Priority: P2
- Confidence: medium
- Impact: Pods may be SIGKILLed before Minecraft finishes stop/save handling under slow or failing RCON conditions.
- Evidence: Defaults include `RCON_RETRIES=5`, `RCON_TIMEOUT=5`, `RCON_RETRY_DELAY=1`, `SHUTDOWN_WAIT_TIMEOUT=90`, `SHUTDOWN_TERM_WAIT=10`, and `SHUTDOWN_SAVE_WAIT_SECONDS=3`. README and examples commonly recommend `terminationGracePeriodSeconds: 120`.
- Why this matters: Kubernetes grace includes `preStop` plus PID 1 signal handling. The boundary doc acknowledges this, but the common examples may still be tight.
- Suggested approach: Document a simple budget formula and consider recommending 180 seconds for any server using RCON preStop, with 120 only for minimal/lightweight cases.
- Acceptance criteria: README and examples explain how to size grace based on RCON retry/timeout settings and world size.
- Risk: Low docs risk; operational impact is positive.
- Suggested labels: docs, kubernetes, shutdown, P2

### 3. Remove or document ignored `MAX_MEMORY` example variable

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Users may believe `MAX_MEMORY` controls JVM heap when it does not.
- Evidence: `entrypoint.sh` defaults `JVM_XMS` and `JVM_XMX`; `scripts/lib/jvm_args.sh` writes `-Xms${JVM_XMS}` and `-Xmx${JVM_XMX}`. Searches show `MAX_MEMORY` only in examples/docs.
- Why this matters: Memory settings are operationally critical in Kubernetes and Docker.
- Suggested approach: Remove `MAX_MEMORY` from examples or explicitly label it as container/platform metadata only if intentionally retained.
- Acceptance criteria: Example memory guidance uses `JVM_XMS`, `JVM_XMX`, and container resource limits consistently.
- Risk: Low.
- Suggested labels: docs, docker, kubernetes, P2
