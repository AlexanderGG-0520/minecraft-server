# Nightly Triage Memo

## Summary

- World reset backups archive literal `/data/world`, even when `LEVEL_NAME` selects and deletes another world directory.
- Nine focused smoke scripts are syntax-checked but never executed in CI.
- Custom mod/config input directories are used for sync but ignored during activation.
- AWS CLI migration lacks real S3-compatible integration coverage and retains obsolete MinIO Client documentation.
- Default shutdown timing can exceed the commonly recommended 120-second Kubernetes grace period.
- Examples still expose an ignored `MAX_MEMORY` variable.

## High Priority Issue Candidates

### 1. Custom world names are not backed up before reset

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Resetting a custom-named world can delete it without backing up that world.
- Evidence: `reset_world` derives its deletion target through `minecraft_world_dir`, respecting `LEVEL_NAME` and `LEVEL`. `create_world_reset_backup` always runs `tar ... -C "${data_dir}" world`. The custom-name smoke disables backups.
- Why this matters: This violates the repository’s strongest world-persistence safety guarantee.
- Suggested approach: Pass the validated selected world name or directory into backup creation and archive that directory.
- Acceptance criteria: With `LEVEL_NAME=custom-world`, the backup contains `custom-world/level.dat`, the unrelated `world/` directory is untouched, and backup failure still prevents deletion.
- Risk: Medium; changes affect destructive reset handling.
- Suggested labels: bug, world-safety, backup, P1

### 2. Existing high-value smoke tests are not executed by CI

- Type: test
- Priority: P1
- Confidence: high
- Impact: Regressions in install ordering, reinstall safety, property mapping, and remove-extra defaults can pass CI.
- Evidence: `.github/workflows/lint-and-smoke.yml` syntax-checks every `test/*.sh`, but does not execute `asset-remove-extra-defaults-smoke.sh`, `install-phase-smoke.sh`, `logging-smoke.sh`, `runtime-marker-validation-smoke.sh`, `runtime-reinstall-mismatch-smoke.sh`, `server-properties-env-detection-smoke.sh`, `server-properties-env-mapping-smoke.sh`, `server-properties-type-smoke.sh`, or `velocity-config-ownership-smoke.sh`.
- Why this matters: Several dormant tests cover persistent-state and configuration safety boundaries.
- Suggested approach: Add explicit workflow steps or a loop over a documented active-test manifest.
- Acceptance criteria: Every active smoke script is executed in pull-request CI; intentionally inactive scripts are clearly identified.
- Risk: Low, although dormant tests may expose existing failures.
- Suggested labels: test, ci, safety, P1

### 3. Custom mod and config staging directories are ignored during activation

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Assets can sync successfully but never reach `/data`, while stale default-directory content may be activated.
- Evidence: `install_mods` uses `INPUT_MODS_DIR`, but `activate_mods` reads `/mods`. `install_configs` uses `INPUT_CONFIG_DIR`, but `activate_configs` reads `/config`.
- Why this matters: Startup logs can report successful S3 synchronization while the runtime receives different content.
- Suggested approach: Resolve each input path once and use it consistently for synchronization and activation, with filesystem safety validation.
- Acceptance criteria: Custom and default input-directory smoke cases activate the exact content that was synchronized.
- Risk: Medium; existing mount conventions may depend on current defaults.
- Suggested labels: bug, s3, persistence, startup, P1

## Medium Priority Issue Candidates

### 1. Add disposable S3-compatible integration coverage

- Type: test
- Priority: P2
- Confidence: high
- Impact: Provider-specific failures involving endpoints, pagination, prefixes, or delete semantics may remain undetected.
- Evidence: Current S3 tests mock `aws`; README advertises MinIO, R2, Garage, B2, and Wasabi compatibility.
- Why this matters: The AWS CLI replacement changed every S3-backed asset and world-install path.
- Suggested approach: Add an opt-in or scheduled workflow against a disposable S3-compatible service.
- Acceptance criteria: Coverage includes list, copy, sync, nested prefixes, custom endpoints, legacy credential mapping, and guarded remove-extra behavior.
- Risk: Medium; requires isolated credentials and guaranteed cleanup.
- Suggested labels: test, s3, aws-cli, integration, migration

### 2. Plugin synchronization leaks temporary files

- Type: bug
- Priority: P2
- Confidence: high
- Impact: Each plugin synchronization can leave an unused temporary file and retains confusing duplicate cleanup state.
- Evidence: `install_plugins` creates `tmp_remote` and `tmp_remote_jars`, then redeclares its temporary variables and replaces the cleanup trap without tracking `tmp_remote_jars`.
- Why this matters: Plugin synchronization is startup-critical and includes deletion-capable behavior.
- Suggested approach: Consolidate temporary allocation and cleanup into one block and one scoped trap.
- Acceptance criteria: No plugin temporary files remain after successful sync, listing failure, or download failure.
- Risk: Medium if mixed with plugin-policy changes; low as a mechanical cleanup.
- Suggested labels: bug, plugins, s3, cleanup, P2

### 3. Kubernetes shutdown guidance understates the possible timeout budget

- Type: docs
- Priority: P2
- Confidence: medium
- Impact: Kubernetes may send SIGKILL before Minecraft finishes saving or stopping.
- Evidence: Defaults include five retries with five-second RCON timeouts, multiple commands, retry delays, a 90-second server-exit wait, and a 10-second TERM wait. README and common examples use 120 seconds.
- Why this matters: Kubernetes includes preStop execution and subsequent PID 1 handling within the same grace period.
- Suggested approach: Document a shutdown-budget formula and recommend at least 180 seconds for modded or large-world servers.
- Acceptance criteria: README and Kubernetes examples explain how RCON settings and world size affect the required grace period.
- Risk: Low.
- Suggested labels: docs, kubernetes, shutdown, rcon, P2

### 4. Safe backup restore procedure is undocumented

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Operators have reset backups but no authoritative procedure for restoring them safely.
- Evidence: Reset backup creation is documented in boundary notes, but no dedicated restore checklist was found.
- Why this matters: Incorrect restoration into a running server or wrong directory can corrupt persistent state.
- Suggested approach: Document a fish-compatible manual restore process covering workload shutdown, archive inspection, preservation of current data, ownership, and restart validation.
- Acceptance criteria: Documentation explicitly prohibits live restoration and covers custom `LEVEL_NAME` archives.
- Risk: Low.
- Suggested labels: docs, backup, restore, operations, P2

### 2. Remove or implement the ignored `MAX_MEMORY` variable

- Type: docs
- Priority: P3
- Confidence: high
- Impact: Operators may believe heap size is configured when it remains at the `JVM_XMX` default.
- Evidence: Multiple Docker and Kubernetes examples set `MAX_MEMORY`; runtime JVM arguments use `JVM_XMS` and `JVM_XMX`.
- Why this matters: Incorrect memory assumptions can cause poor performance or container OOM termination.
- Suggested approach: Replace `MAX_MEMORY` with `JVM_XMX`, or deliberately implement and document a compatibility alias.
- Acceptance criteria: All examples use supported heap variables and agree with container memory limits.
- Risk: Low.
- Suggested labels: docs, examples, jvm, memory, P3

### 3. Clarify the unused resourcepack activation helper

- Type: design
- Priority: P3
- Confidence: medium
- Impact: Future changes may accidentally activate synchronized files into `/data/resourcepacks`, contrary to documented behavior.
- Evidence: `activate_resourcepacks` exists and is source-smoked, but the install phase intentionally calls only `install_resourcepacks`.
- Why this matters: Local object synchronization and Minecraft client distribution are separate concerns.
- Suggested approach: Mark the helper as reserved/unused or remove it in a dedicated cleanup after confirming no external consumers.
- Acceptance criteria: Code and documentation clearly state whether local activation is supported.
- Risk: Low.
- Suggested labels: design, resourcepacks, cleanup, P3

### 4. Record AWS CLI version expectations across image flavors

- Type: design
- Priority: P3
- Confidence: medium
- Impact: CPU and GPU images may behave differently with the same S3 configuration.
- Evidence: CPU images install Debian’s `awscli`; the GPU image downloads AWS CLI v2 directly.
- Why this matters: S3 compatibility should not depend unexpectedly on runtime flavor.
- Suggested approach: Document the divergence or standardize packaging after compatibility evaluation.
- Acceptance criteria: Published image documentation identifies supported AWS CLI major versions and relevant compatibility expectations.
- Risk: Medium if packaging is standardized; low for documentation.
- Suggested labels: design, docker, aws-cli, compatibility, P3

## Suspicious Areas That Need Human Judgment

- `rcon-stop` intentionally exits successfully when RCON fails so Kubernetes continues termination; PID 1 provides the fallback. Changing this would alter operational semantics.
- The 30-second cross-process RCON lock wait consumes additional termination grace when preStop and PID 1 overlap.
- World installation accepts exactly one direct-child ZIP under a prefix. This is conservative, but operators may expect version selection.
- CPU and GPU AWS CLI implementations could differ on less common S3-compatible providers; no concrete failure was found.
- The shutdown boundary document contains stale ownership statements about functions that have already moved into library files.
