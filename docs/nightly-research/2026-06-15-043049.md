# Nightly Triage Memo

## Summary

- World reset backups appear unsafe for custom `LEVEL_NAME`: reset deletes the configured world directory, but backup archives literal `world`.
- Datapack precedence is ambiguous: S3 sync writes into the world, then `/datapacks` activation can replace it.
- PR CI only builds `runtime-jre21`; recent GPU AWS CLI packaging changes are mostly caught after merge.
- Kubernetes examples use `:latest`, but publish workflow emits target tags such as `runtime-jre21`.
- Several MinIO `mc` remediation docs are stale after the AWS CLI migration.
- Manual release backfill can tag the currently checked-out ref as an older release tag.

## High Priority Issue Candidates

### 1. Custom `LEVEL_NAME` world reset backup archives the wrong directory

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Operators using `LEVEL_NAME` can delete `/data/<LEVEL_NAME>` while the backup captures `/data/world`, potentially leaving no backup of the deleted world.
- Evidence: `reset_world` derives `world_dir` through `minecraft_world_dir`; `create_world_reset_backup` runs `tar -czf ... -C "${data_dir}" world`.
- Why this matters: World reset is destructive, and the project philosophy treats world data and mounted volumes as user-owned state.
- Suggested approach: Archive the actual selected world basename from `minecraft_world_name` and add custom-level backup smoke coverage.
- Acceptance criteria: With `LEVEL_NAME=custom-world` and backup enabled, reset produces an archive containing `custom-world/level.dat`; backup failure still prevents deletion.
- Risk: Medium; touches destructive reset behavior.
- Suggested labels: bug, world-safety, backup, P1

### 2. S3 datapack sync can be replaced by local `/datapacks` activation

- Type: bug
- Priority: P1
- Confidence: medium
- Impact: S3 datapacks can be silently overwritten by mounted/input datapacks during the same install phase.
- Evidence: `install_phase.sh` calls `install_datapacks` then `activate_datapacks`; the first writes to `${DATA_DIR}/world/datapacks`, the second activates `/datapacks` into the same destination.
- Why this matters: GitOps users may expect S3 to be authoritative after sync.
- Suggested approach: Decide intended precedence, then make S3 and mounted datapack behavior consistent.
- Acceptance criteria: A smoke test covers S3 datapack sync plus non-empty `/datapacks` and proves the chosen precedence.
- Risk: Medium; existing deployments may depend on current ordering.
- Suggested labels: bug, s3, datapacks, lifecycle, P1

### 3. PR CI does not build the GPU runtime that recently changed AWS CLI packaging

- Type: infra
- Priority: P1
- Confidence: high
- Impact: `runtime-jre25-gpu` packaging regressions can merge before publish catches them.
- Evidence: `runtime-smoke.yml` builds only `runtime-jre21`; recent history includes GPU AWS CLI fixes, while GPU uses a different base/package path.
- Why this matters: GPU runtime is advertised and S3 sync depends on AWS CLI availability.
- Suggested approach: Add PR smoke coverage for `runtime-jre25-gpu`, or a lightweight packaging check for `aws --version`, `/entrypoint.sh`, and sourced libraries.
- Acceptance criteria: Pull requests fail before merge if `runtime-jre25-gpu` lacks AWS CLI or entrypoint dependencies.
- Risk: Low to medium; CI time may increase.
- Suggested labels: infra, ci, gpu, s3, P1

### 3. Manual release backfill can publish current code under an older release tag

- Type: release
- Priority: P2
- Confidence: medium
- Impact: A workflow_dispatch backfill for `vX.Y.Z` can build the selected ref instead of the matching tag, then publish images named as that release.
- Evidence: `publish.yml` accepts `release_tag`, validates its format, and uses it only for image tag names; checkout still uses the workflow dispatch ref.
- Why this matters: Versioned container tags should correspond to the source release they claim.
- Suggested approach: For `release_tag`, checkout that tag explicitly or fail unless `github.ref_name` matches it.
- Acceptance criteria: Manual backfill cannot publish images for `vX.Y.Z` unless the Docker build source is exactly `refs/tags/vX.Y.Z`.
- Risk: Medium; release flow changes need careful validation.
- Suggested labels: release, github-actions, supply-chain, P2

### 4. Backup exists, but restore workflow is under-documented

- Type: docs
- Priority: P2

- Evidence: README and docs cover reset/install safety, but no dedicated restore checklist was found.
- Why this matters: Backup safety is incomplete without recovery guidance.
- Suggested approach: Document fish-compatible manual restore steps: stop workload, preserve current data, extract selected archive, validate ownership, then restart.
- Acceptance criteria: Docs include a restore checklist and warn against restoring into a live server.
- Risk: Low.
- Suggested labels: docs, backup, restore, ops, P2
