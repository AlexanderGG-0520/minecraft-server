# Nightly Triage Memo

## Summary

- Generic S3 `*_REMOVE_EXTRA=true` safety checks only prove the prefix has at least one object, not that it has meaningful files for the target asset type.
- Datapacks have a confusing dual path: S3 sync writes directly into `/data/world/datapacks`, then activation can replace that directory from `/datapacks`.
- Custom staging directory drift remains for mods/configs: install can use `INPUT_*_DIR`, while activation still reads hard-coded defaults.
- Runtime smoke now includes a `runtime-jre25-gpu` packaging job, reducing a previously noted GPU AWS CLI coverage gap.
- Several stale MinIO `mc` docs still describe obsolete implementation details after the AWS CLI migration.
- Release backfill via `workflow_dispatch.release_tag` appears to tag the workflow checkout, not necessarily the requested release tag.

## High Priority Issue Candidates

### 1. Strengthen generic S3 remove-extra non-empty checks

- Type: bug
- Priority: P1
- Confidence: medium
- Impact: `MODS_REMOVE_EXTRA`, `CONFIGS_REMOVE_EXTRA`, `DATAPACKS_REMOVE_EXTRA`, or `RESOURCEPACKS_REMOVE_EXTRA` can proceed when the remote prefix only contains placeholder or directory-marker objects.
- Evidence: `ensure_s3_source_nonempty_for_remove` calls `s3_list_paths` and only checks that the temp listing file is non-empty. `s3_list_paths` prints every returned key. Generic asset syncs then call `aws s3 sync --delete`.
- Why this matters: The project explicitly treats mounted `/data` and persistent asset state as user-owned. A placeholder-only prefix can look non-empty while still being effectively empty for a mirror sync.
- Suggested approach: Make remove-extra validation feature-aware, such as requiring at least one non-directory object after trimming the selected prefix. For mods/plugins, prefer top-level `.jar` checks; for configs/datapacks/resourcepacks, ignore keys ending in `/` and optionally allow a documented sentinel file.
- Acceptance criteria: Smoke tests cover placeholder-only prefixes and prove remove-extra refuses to prune local state.
- Risk: Medium; stricter checks may block existing operators who intentionally mirror empty prefixes.
- Suggested labels: bug, s3, persistence, safety, P1

### 2. Resolve datapack S3 sync versus activation path conflict

- Type: design
- Priority: P1
- Confidence: high
- Impact: A pod with both `DATAPACKS_S3_BUCKET` and a non-empty `/datapacks` mount can sync S3 datapacks into `/data/world/datapacks`, then replace them from `/datapacks`.
- Evidence: `install_phase.sh` calls `install_datapacks` and then `activate_datapacks`. `install_datapacks` writes to `${DATA_DIR}/world/datapacks`; `activate_datapacks` activates `/datapacks` into the same destination.
- Why this matters: Datapacks are world-coupled state. Surprising overwrite order can affect gameplay behavior and world persistence.
- Suggested approach: Pick one model: S3 sync to staging then activate, or direct-to-world with no activation when S3 is configured. Document precedence if both sources are intentionally supported.
- Acceptance criteria: Tests prove the chosen precedence when S3 and `/datapacks` are both present.
- Risk: Medium; changing order may affect existing deployments that rely on current precedence.
- Suggested labels: design, bug, datapacks, s3, P1

### 3. Fix or explicitly de-support custom mods/config staging directories

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Operators setting `INPUT_MODS_DIR` or `INPUT_CONFIG_DIR` can sync into one directory while activation reads another.
- Evidence: `install_mods` writes to `${INPUT_MODS_DIR}`, but `activate_mods` reads `/mods`. `install_configs` writes to `${INPUT_CONFIG_DIR:-/config}`, but `activate_configs` reads `/config`.
- Why this matters: Asset activation touches persistent `/data` and should not silently use stale/default staging content.
- Suggested approach: Either make activation use the same resolved input directory, or mark those variables internal and refuse non-default values.
- Acceptance criteria: Smoke tests cover custom mods/config staging end-to-end.
- Risk: Medium because mount conventions may be in use.
- Suggested labels: bug, s3, persistence, P1

## Medium Priority Issue Candidates

### 1. Make release backfill checkout semantics explicit

- Type: release
- Priority: P2
- Confidence: medium
- Impact: Manual backfill can publish `vX.Y.Z-*` image tags from whatever ref the workflow was dispatched on, not necessarily from the requested release tag.
- Evidence: `publish.yml` validates `inputs.release_tag` and uses it when preparing image tags, but checkout is the default workflow ref.
- Why this matters: Versioned container tags should map predictably to source revisions.
- Suggested approach: On `workflow_dispatch` with `release_tag`, explicitly checkout that tag or fail unless `github.ref_name` matches the requested tag.
- Acceptance criteria: A manual dispatch for `release_tag=v1.2.3` builds exactly the `v1.2.3` source tree.
- Risk: Low.
- Suggested labels: release, infra, P2

### 2. Archive or update stale MinIO `mc` remediation docs

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Maintainers may follow obsolete guidance describing `mc-builder`, `/usr/local/bin/mc`, `configure_mc_alias`, and `mc mirror`.
- Evidence: Current `Dockerfile` installs AWS CLI, and `scripts/lib/s3_client.sh` wraps `aws`. Several `docs/minio-mc-*` and world-install boundary docs still describe MinIO Client as current.
- Why this matters: The project philosophy prefers documentation that describes real operational behavior.
- Suggested approach: Add historical headers or rewrite the docs around AWS CLI migration status.
- Acceptance criteria: Searches for `mc-builder`, `configure_mc_alias`, and `/usr/local/bin/mc` clearly identify historical notes.
- Risk: Low.
- Suggested labels: docs, s3, migration, P2

### 3. Add in-image mocked AWS CLI behavior smoke

- Type: test
- Priority: P2
- Confidence: medium
- Impact: Local mocked tests validate helpers, but fewer tests prove the built image executes the same helper paths with packaged dependencies.
- Evidence: `test/s3-client-smoke.sh` mocks `aws` outside the image. Runtime smoke checks image syntax and GPU `aws --version`, but does not run a CPU/GPU in-image helper flow for endpoint/env mapping.
- Why this matters: AWS CLI replacement remains a migration risk for S3-compatible storage.
- Suggested approach: In runtime smoke, place a mocked `aws` earlier in `PATH`, source `s3_client.sh`, and assert endpoint args, legacy credential mapping, `s3 cp`, `s3 sync`, and `s3api list-objects-v2`.
- Acceptance criteria: CPU and GPU runtime smoke both exercise one in-container mocked S3 helper path.
- Risk: Low.
- Suggested labels: test, s3, docker, P2

## Suspicious Areas That Need Human Judgment

- Generic S3 remove-extra may intentionally allow empty mirrors; if so, the current docs should make that risk explicit.
- `MODPACK_FORCE_REINSTALL=true` can replace previously managed files but still refuses user-owned jars; this seems safe, but operational expectations may differ.
- CPU images use distro `awscli` while GPU uses AWS CLI v2; no breakage found, but behavior drift across providers remains possible.
- World install allows exactly one top-level zip under the prefix; safe, but operators may expect newest-by-name behavior.
- `reset-world.flag` expiry logs an error and consumes the flag; this may be intentional but should match maintainer expectations.
