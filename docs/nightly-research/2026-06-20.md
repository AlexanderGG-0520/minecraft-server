# Nightly Triage Memo

## Summary

- Custom `LEVEL_NAME` remains unsafe during reset backup and is ignored by datapack handling.
- World replacement deletes the existing world before a cross-filesystem move, creating a data-loss window.
- Image publishing runs independently of CI success and can publish broken `main` commits.
- Several high-value smoke scripts exist but are only syntax-checked.
- Custom asset input directories are inconsistently honored during activation.
- AWS CLI migration coverage remains primarily mocked and does not verify real S3-compatible behavior.

## High Priority Issue Candidates

### 1. World reset backs up the wrong directory with custom `LEVEL_NAME`

- Type: bug
- Priority: P1
- Confidence: high
- Impact: A custom world can be deleted without a valid backup of that world.
- Evidence: `reset_world` derives its deletion target through `minecraft_world_dir`, but `create_world_reset_backup` always archives the literal `world` directory using `tar -C "${data_dir}" world`.
- Why this matters: If both `/data/world` and `/data/<LEVEL_NAME>` exist, reset can successfully archive the former and then delete the latter.
- Suggested approach: Pass the validated world basename into the backup helper and archive that exact directory.
- Acceptance criteria: A reset with `LEVEL_NAME=custom-world` backs up and deletes only `custom-world`; tests cover both presence and absence of `/data/world`.
- Risk: High because the change touches destructive reset behavior.
- Suggested labels: bug, world-safety, backup, P1

### 2. World installation removes the existing world before replacement is durable

- Type: design
- Priority: P1
- Confidence: high
- Impact: A failed replacement can leave the persistent volume without its previous world or with a partial new world.
- Evidence: `install_world` calls `safe_rm_rf "${WORLD_DIR}"` before moving the extracted directory from `/tmp` into `/data`. These paths are commonly different filesystems in containers, so `mv` becomes a non-atomic copy-and-delete operation.
- Why this matters: World installation is explicitly identified as a high-risk persistent-state operation.
- Suggested approach: Stage the extracted world inside the validated data volume, then rename the old and staged directories with rollback handling.
- Acceptance criteria: Simulated staging or rename failure preserves the original world and reset flag; successful replacement leaves no staging directories.
- Risk: High; filesystem and rollback behavior require focused testing.
- Suggested labels: bug, world-safety, persistence, P1

### 3. Publish workflow is not gated on lint or runtime smoke success

- Type: release
- Priority: P1
- Confidence: high
- Impact: A commit pushed to `main` can publish all mutable runtime tags even when another CI workflow fails.
- Evidence: `publish.yml`, `lint-and-smoke.yml`, and `runtime-smoke.yml` independently trigger on pushes to `main`; publishing has no dependency on either validation workflow.
- Why this matters: Broken startup, shutdown, or persistence behavior can reach both GHCR and Docker Hub immediately.
- Suggested approach: Publish through a workflow that runs only after required validation succeeds, or perform required validation within the publishing workflow.
- Acceptance criteria: Intentionally failing required smoke validation prevents all image pushes.
- Risk: Medium; workflow dependency and tag-event behavior must be preserved.
- Suggested labels: release, ci, docker, P1

### 4. Datapack handling ignores custom world names

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Datapacks can be installed into an unused `/data/world` while the server runs `/data/<LEVEL_NAME>`.
- Evidence: `install_datapacks` uses `${DATA_DIR}/world/datapacks`, and `activate_datapacks` uses `${DATA_DIR}/world`; other world operations use `minecraft_world_dir`.
- Why this matters: Operators may believe a datapack is active when it was written into the wrong world.
- Suggested approach: Resolve the world directory through the shared world-path helper everywhere.
- Acceptance criteria: Custom `LEVEL_NAME` smoke coverage verifies sync, activation, and default-world compatibility.
- Risk: Medium because existing deployments may already contain files under the incorrect path.
- Suggested labels: bug, datapacks, world-paths, P1

### 5. Existing high-value smoke scripts are not executed by CI

- Type: test
- Priority: P1
- Confidence: high
- Impact: Regressions in install ordering, server property mapping, reinstall safety, and remove-extra defaults can merge unnoticed.
- Evidence: CI syntax-checks every `test/*.sh`, but direct execution is absent for scripts including `install-phase-smoke.sh`, `asset-remove-extra-defaults-smoke.sh`, `server-properties-env-mapping-smoke.sh`, and `runtime-reinstall-mismatch-smoke.sh`.
- Why this matters: These tests cover declared safety boundaries and already exist at minimal additional implementation cost.
- Suggested approach: Execute all active smoke scripts explicitly and document any intentionally local-only tests.
- Acceptance criteria: Every active smoke script runs in CI, and a deliberate assertion failure fails the workflow.
- Risk: Low, although dormant tests may expose existing drift.
- Suggested labels: test, ci, P1

## Medium Priority Issue Candidates

### 1. Custom asset input directories are ignored during activation

- Type: bug
- Priority: P2
- Confidence: high
- Impact: Assets can sync into a configured staging directory but activate from a stale default directory.
- Evidence: Mods and configs install through `INPUT_MODS_DIR` and `INPUT_CONFIG_DIR`, while activation reads hard-coded `/mods` and `/config`. `INPUT_DATAPACKS_DIR` is declared but unused.
- Why this matters: Mounted input directories are part of container and GitOps operation.
- Suggested approach: Decide whether these are supported configuration variables, then use the same resolved paths for install and activation.
- Acceptance criteria: End-to-end tests cover custom and default input paths.
- Risk: Medium due to existing mount conventions.
- Suggested labels: bug, assets, persistence, P2

### 2. Add real S3-compatible migration coverage

- Type: test
- Priority: P2
- Confidence: high
- Impact: Provider-specific AWS CLI failures may not be detected by mocked command tests.
- Evidence: Current tests mock `aws`; advertised providers include MinIO, R2, Garage, B2, and Wasabi. No integration test verifies endpoint addressing, pagination, sync deletion, or exact object download.
- Why this matters: The AWS CLI replacement affects world installation and all S3-backed assets.
- Suggested approach: Add a scheduled or manual workflow using disposable object storage and non-production buckets.
- Acceptance criteria: Coverage includes list, copy, sync, nested prefixes, empty-source deletion refusal, legacy credentials, and custom endpoints.
- Risk: Medium due to service setup and secret handling.
- Suggested labels: test, s3, aws-cli, migration, P2

### 3. Validate all shutdown timing and retry environment variables

- Type: bug
- Priority: P2
- Confidence: medium
- Impact: Invalid numeric values can cause arithmetic errors or unpredictable signal handling during termination.
- Evidence: `SHUTDOWN_SAVE_WAIT_SECONDS` is validated, but values such as `STOP_SERVER_ANNOUNCE_DELAY`, `RCON_RETRIES`, `RCON_RETRY_DELAY`, `RCON_TIMEOUT`, and shutdown wait values are used without equivalent preflight validation.
- Why this matters: Shutdown is the last opportunity to save persistent world state.
- Suggested approach: Validate non-negative integer timing values during preflight and add malformed-value smoke cases.
- Acceptance criteria: Invalid values fail before runtime starts with a specific variable name.
- Risk: Low to medium; stricter validation may reject previously tolerated configurations.
- Suggested labels: bug, shutdown, rcon, validation, P2

### 4. Plugin sync allocates an untracked temporary file

- Type: bug
- Priority: P2
- Confidence: high
- Impact: Each plugin synchronization can leak a temporary file for the container lifetime.
- Evidence: `install_plugins` creates `tmp_remote` and `tmp_remote_jars`, then redeclares its temp variables and cleanup trap; the first `tmp_remote_jars` is no longer tracked.
- Why this matters: The duplication also makes deletion-capable plugin synchronization harder to audit.
- Suggested approach: Consolidate temporary allocation and cleanup into one block without changing plugin policy.
- Acceptance criteria: Success and failure tests leave no plugin-listing temporary files.
- Risk: Low if kept mechanical.
- Suggested labels: bug, plugins, cleanup, P2

### 5. Document restore and backup retention operations

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Operators can create reset backups but lack a defined safe restore procedure or retention policy.
- Evidence: Reset writes timestamped archives under `/data/backups`; no automated retention or complete restore guide is present.
- Why this matters: Unlimited backups can exhaust a PVC, while an improvised restore can overwrite an active world.
- Suggested approach: Document fish-compatible stop, verification, extraction, ownership, and rollback steps, plus retention expectations.
- Acceptance criteria: Documentation clearly covers restore safety and states whether retention is manual.
- Risk: Low.
- Suggested labels: docs, backup, restore, operations, P2

## Suspicious Areas That Need Human Judgment

- CPU images use Debian’s `awscli`, while the GPU image downloads upstream AWS CLI v2; behavior and update cadence may differ.
- The GPU AWS CLI download and `mcrcon` binary download are not checksum-pinned.
- `activate_resourcepacks` exists but is intentionally absent from the install phase according to current README behavior.
- Mutable runtime tags use `imagePullPolicy: Always` in some examples and `IfNotPresent` in others, which creates different rollout semantics.
- World backup archives have no collision protection beyond one-second timestamp resolution.
- `is_world_generated` and `wait_for_worldgen` still reference `/data/world`, but currently appear unused.
