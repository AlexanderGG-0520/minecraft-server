# Nightly Triage Memo

## Summary

- Recent commits fixed several previously identified world path, datapack, plugin sync, and Forge bootstrap risks.
- The clearest remaining operational gap is release safety: `publish.yml` can publish independently of the lint/runtime smoke workflows.
- AWS CLI migration behavior is well covered with mocks, but not with a real disposable S3-compatible endpoint.
- CPU and GPU images install AWS CLI from different sources, which may produce subtly different S3 behavior.
- Several `docs/minio-mc-*` files still describe MinIO Client implementation details as current, despite the Dockerfile and helpers now using AWS CLI.
- Several committed smoke tests are syntax-checked but do not appear to run as CI behavior checks.
- Example docs still include `MAX_MEMORY`, which appears ignored by runtime heap generation.

## High Priority Issue Candidates

### 1. Publish workflow can push images without being gated on smoke checks

- Type: release
- Priority: P1
- Confidence: medium
- Impact: A `main` or `v*` push can publish production-facing image tags even if lint or runtime smoke would fail for the same commit.
- Evidence: `.github/workflows/publish.yml` runs directly on `push` to `main`, `v*`, and `workflow_dispatch`; lint/static smoke and runtime smoke are separate workflows.
- Why this matters: The Docker tags are the project’s primary release artifact, and this repository optimizes for predictable operational behavior.
- Suggested approach: Gate publishing on successful required workflows for the exact SHA, or move image publication behind a `workflow_run` release gate.
- Acceptance criteria: A failing lint or runtime smoke workflow prevents all image pushes for that commit, including versioned tags.
- Risk: Medium; workflow-event wiring and tag releases need careful testing.
- Suggested labels: release, ci, docker, reliability, P1

### 2. AWS CLI migration lacks real S3-compatible integration coverage

- Type: test
- Priority: P1
- Confidence: high
- Impact: Mocked `aws` tests may miss provider behavior around endpoint URLs, credentials, pagination, delete semantics, path handling, and sync behavior.
- Evidence: `test/s3-client-smoke.sh`, resourcepack tests, plugin tests, and workflow inline tests mock `aws`; README advertises AWS S3, MinIO, R2, Garage, B2, and Wasabi.
- Why this matters: S3-backed assets and world archives are core startup behavior, and the repo recently moved away from MinIO Client.
- Suggested approach: Add a non-default scheduled/manual workflow against disposable MinIO or another controlled S3-compatible endpoint.
- Acceptance criteria: The workflow validates list, cp, sync, endpoint URL, credential mapping, remove-extra safety checks, and delete behavior without production credentials.
- Risk: Medium; integration cleanup and secret boundaries must be explicit.
- Suggested labels: test, s3, aws-cli, migration, P1

## Medium Priority Issue Candidates

### 1. CPU and GPU images use different AWS CLI implementations

- Type: infra
- Priority: P2
- Confidence: high
- Impact: S3 behavior may differ by runtime flavor.
- Evidence: CPU runtime-base installs Debian `awscli`; `runtime-jre25-gpu` downloads AWS CLI v2 from Amazon. CI checks `aws --version` for GPU but does not compare S3 behavior across image families.
- Why this matters: Operators should not see object-storage behavior change when switching runtime targets.
- Suggested approach: Add image-level smoke tests for both CPU and GPU targets that exercise the same AWS CLI helper flows with a mocked or disposable endpoint.
- Acceptance criteria: Both image families validate endpoint args, legacy env var mapping, region defaults, cp, list, sync, and remove-extra conversion.
- Risk: Low to medium.
- Suggested labels: infra, test, docker, gpu, s3, P2

### 2. Stale MinIO Client docs conflict with current AWS CLI implementation

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Maintainers may follow obsolete remediation guidance for `mc-builder`, `MC_RELEASE`, `/usr/local/bin/mc`, `configure_mc_alias`, or `mc mirror`.
- Evidence: Dockerfile now installs `awscli`; `scripts/lib/s3_client.sh` wraps `aws`; multiple `docs/minio-mc-*` files still describe MinIO Client as current.
- Why this matters: Security and dependency remediation docs should point maintainers toward the current implementation.
- Suggested approach: Add archival banners to old MinIO Client docs or rewrite them as historical records with pointers to AWS CLI docs.
- Acceptance criteria: Searches for old MinIO Client terms clearly distinguish historical notes from current behavior.
- Risk: Low.
- Suggested labels: docs, s3, migration, P2

### 3. Some committed smoke tests are not executed in CI

- Type: test
- Priority: P2
- Confidence: medium
- Impact: Useful regression tests can drift because they are only syntax-checked.
- Evidence: `.github/workflows/lint-and-smoke.yml` runs a selected set of `test/*.sh`; files such as `asset-remove-extra-defaults-smoke.sh`, `install-phase-smoke.sh`, `runtime-reinstall-mismatch-smoke.sh`, `server-properties-env-mapping-smoke.sh`, and `velocity-config-ownership-smoke.sh` do not appear as behavioral CI steps.
- Why this matters: These tests cover safety-sensitive defaults, install ordering, reinstall preservation, secrets masking, and Velocity config ownership.
- Suggested approach: Run all active `test/*.sh` files explicitly or maintain a documented exclusion list.
- Acceptance criteria: Every committed smoke test either runs in CI or has a stated reason for exclusion.
- Risk: Low; enabling dormant tests may reveal existing drift.
- Suggested labels: test, ci, P2

### 4. README CI coverage summary is outdated

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Maintainers and contributors may underestimate or misunderstand what CI actually verifies.
- Evidence: README says lint/static smoke runs `bash -n` and ShellCheck; the workflow now runs many focused smokes for S3, resourcepacks, plugins, world install/reset, RCON, Modrinth, runtime marker handling, and more.
- Why this matters: Accurate docs help contributors choose the right checks before changing high-risk startup paths.
- Suggested approach: Update the CI section to describe static checks, shell helper smokes, image runtime smoke, GPU packaging smoke, and known non-goals.
- Acceptance criteria: README matches current workflow responsibilities without overstating production coverage.
- Risk: Low.
- Suggested labels: docs, ci, P2

## Low Priority / Backlog Candidates

### 1. Remove or implement ignored `MAX_MEMORY` examples

- Type: docs
- Priority: P3
- Confidence: high
- Impact: Operators may believe heap size is configured by `MAX_MEMORY` when runtime heap generation uses `JVM_XMS` and `JVM_XMX`.
- Evidence: Examples still set `MAX_MEMORY`; `scripts/lib/jvm_args.sh` writes `-Xms${JVM_XMS}` and `-Xmx${JVM_XMX}`.
- Why this matters: Incorrect memory assumptions can lead to poor scheduling or OOM behavior.
- Suggested approach: Replace `MAX_MEMORY` examples with `JVM_XMX`, or implement and document `MAX_MEMORY` as a compatibility alias.
- Acceptance criteria: Examples contain only effective memory variables, or docs explain the alias behavior.
- Risk: Low.
- Suggested labels: docs, examples, jvm, P3

### 2. Clarify intentionally unused `activate_resourcepacks`

- Type: design
- Priority: P3
- Confidence: high
- Impact: Future maintainers may accidentally call it and change `/data/resourcepacks` persistence behavior.
- Evidence: `activate_resourcepacks` exists and is source-smoked; `install_phase.sh` intentionally calls `install_resourcepacks` but not `activate_resourcepacks`; README says local resourcepack sync is not activated into `/data/resourcepacks`.
- Why this matters: Resourcepack handling has distinct local-validation and client-distribution semantics.
- Suggested approach: Add a short code comment or docs note explaining whether the helper is reserved for future local serving or should be removed.
- Acceptance criteria: Tests and docs make the non-activation behavior explicit.
- Risk: Low.
- Suggested labels: design, docs, resourcepacks, P3

### 3. Reduce Dockerfile runtime-stage duplication

- Type: refactor
- Priority: P3
- Confidence: high
- Impact: Repeated Java runtime stage blocks increase maintenance risk when changing users, directories, entrypoint permissions, or installed binaries.
- Evidence: `runtime-jre8`, `runtime-jre11`, `runtime-jre17`, `runtime-jre21`, and `runtime-jre25` repeat nearly identical setup.
- Why this matters: Docker image compatibility changes should be made once where possible.
- Suggested approach: Extract common setup into a shared stage pattern if it can be done without changing image behavior.
- Acceptance criteria: Produced image layout, user, workdir, entrypoint, and copied files remain equivalent.
- Risk: Medium; Dockerfile refactors can change cache and layer behavior.
- Suggested labels: refactor, docker, P3
