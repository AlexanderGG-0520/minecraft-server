# Nightly Triage Memo

## Summary

- RCON shutdown has a live validation gap: `STOP_SERVER_ANNOUNCE_DELAY=invalid` causes an unbound-variable arithmetic error before save/stop commands run.
- Datapack install order can make S3 datapacks disappear when `/datapacks` is also non-empty.
- Several existing smoke scripts are still syntax-checked but not executed by CI, including install-order, marker, reinstall, and `server.properties` mapping tests.
- Some MinIO `mc` remediation docs still describe `mc` as current behavior even though runtime code and Dockerfile now use AWS CLI.
- Focused read-only/temp-only checks passed for syntax, S3 client, resourcepacks, world install, world reset, datapacks path, and RCON smoke.

## High Priority Issue Candidates

### 1. RCON shutdown rejects invalid announce delay with shell error before stop flow

- Type: bug
- Priority: P1
- Confidence: high
- Impact: A bad `STOP_SERVER_ANNOUNCE_DELAY` value can interrupt graceful shutdown before `citizens save`, `save-all`, and `stop`.
- Evidence: `scripts/lib/rcon.sh:86` reads `STOP_SERVER_ANNOUNCE_DELAY`, but only `SHUTDOWN_SAVE_WAIT_SECONDS` is validated at `scripts/lib/rcon.sh:91`. Arithmetic evaluation happens at `scripts/lib/rcon.sh:102`. Ad hoc temp-only reproduction returned `line 102: invalid: unbound variable`.
- Why this matters: Shutdown is one of the repository’s highest-risk paths, and bad env should fail clearly or fall back safely rather than skipping Minecraft-aware stop.
- Suggested approach: Validate `STOP_SERVER_ANNOUNCE_DELAY` as a non-negative integer before arithmetic and log a clear `[shutdown] invalid ...` message.
- Acceptance criteria: A smoke test with `STOP_SERVER_ANNOUNCE_DELAY=invalid` exits predictably with a clear log and does not produce a Bash unbound-variable error.
- Risk: Low; scoped validation change in RCON shutdown path.
- Suggested labels: bug, shutdown, rcon, P1

### 2. S3 datapack sync can be overwritten by local `/datapacks` activation

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Operators using both `DATAPACKS_S3_BUCKET` and a non-empty `/datapacks` input can see freshly synced S3 datapacks replaced during the same install phase.
- Evidence: `install_datapacks` syncs S3 directly into `${world_dir}/datapacks` at `scripts/lib/content_assets.sh:434` and `scripts/lib/content_assets.sh:457`. `activate_datapacks` later calls `activate_dir "/datapacks" "${world_dir}/datapacks"` at `scripts/lib/content_assets.sh:475`. Install order runs `install_datapacks` then `activate_datapacks` at `scripts/lib/install_phase.sh:22-23`. A temp-only reproduction showed `s3_exists=no local_exists=yes` after activation.
- Why this matters: GitOps users can unintentionally lose the S3-authoritative datapack set because a mounted local input directory silently wins.
- Suggested approach: Define precedence explicitly. Either stage S3 into the same input activation flow, skip local activation when S3 datapacks are configured, or fail fast when both sources are non-empty.
- Acceptance criteria: CI covers the mixed-source case and the chosen behavior is documented.
- Risk: Medium; datapack precedence may be user-visible.
- Suggested labels: bug, datapacks, s3, world-safety, P1

### 2. MinIO `mc` docs still describe obsolete current behavior after AWS CLI migration

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Maintainers may follow stale remediation guidance or preserve nonexistent `mc` behavior during future S3 work.
- Evidence: `Dockerfile:32-35` installs `awscli`, GPU runtime installs AWS CLI at `Dockerfile:194-205`, and `scripts/lib/s3_client.sh:23-92` wraps `aws`. But `docs/minio-mc-remediation-boundary.md:19-53` still says the image currently uses MinIO `mc`, has an `mc-builder` stage, and uses `mc alias`, `mc find`, and `mc mirror`.
- Why this matters: The project has explicitly moved to AWS CLI-compatible S3 behavior; stale docs increase migration and security-remediation risk.
- Suggested approach: Mark old MinIO docs as historical, or rewrite them to say they predate AWS CLI migration and point to `docs/s3-client-cleanup-boundary.md`.
- Acceptance criteria: No doc page presents `mc` as current runtime behavior unless it is clearly historical.
- Risk: Low; docs-only cleanup.
- Suggested labels: docs, s3, aws-cli, migration, P2

### 3. `server.properties` support for Quilt/Mohist/Taiyitist/Youer is unclear

- Type: design
- Priority: P2
- Confidence: medium
- Impact: Some supported `server.jar` runtimes may skip `server.properties` bootstrap and env diff even though they likely consume Minecraft server properties.
- Evidence: Runtime support includes `mohist`, `quilt`, `taiyitist`, and `youer` in `scripts/lib/runtime.sh:11` and launch path at `scripts/lib/runtime_launch.sh:51`, but `uses_server_properties` includes only vanilla, paper, purpur, spigot, fabric, forge, and neoforge at `scripts/lib/runtime.sh:32-37`.
- Why this matters: Operators may expect standard env mappings like `ONLINE_MODE`, `SERVER_PORT`, and `ENABLE_RCON` to apply consistently across supported server runtimes.
- Suggested approach: Decide whether these runtimes should use `server.properties`; if yes, add them with smoke coverage. If no, document the exclusion.
- Acceptance criteria: Each supported runtime has documented `server.properties` behavior and tests cover at least the classification helper.
- Risk: Medium; changing bootstrap behavior may affect existing users.
- Suggested labels: design, server-properties, runtime, P2

## Low Priority / Backlog Candidates

### 1. Safe backup restore procedure is still under-documented

- Type: docs
- Priority: P3
- Confidence: medium
- Impact: Operators can create reset backups but lack an authoritative restore sequence.
- Evidence: `world_reset.sh` creates timestamped archives under `${DATA_DIR}/backups`, and docs describe reset/backup internals, but no dedicated operator restore runbook was found.
- Why this matters: Improvised restore can overwrite a live world or restore the wrong `LEVEL_NAME`.
- Suggested approach: Add fish-compatible restore guidance covering workload shutdown, archive inspection, current-data preservation, extraction target, ownership, and restart validation.
- Acceptance criteria: Docs warn against restoring into a live server and include a safe restore checklist.
- Risk: Low; docs-only.
- Suggested labels: docs, backup, restore, kubernetes, P3

### 2. `activate_resourcepacks` remains defined but intentionally unused

- Type: refactor
- Priority: P3
- Confidence: medium
- Impact: Future maintainers may assume local resourcepack activation is part of install flow.
- Evidence: `activate_resourcepacks` exists in `scripts/lib/content_assets.sh`, is declared in source smokes, but `scripts/lib/install_phase.sh:30-32` calls only `install_resourcepacks`. README says resourcepacks are synced locally for validation/inspection and not activated.
- Why this matters: Resourcepack behavior affects client-facing URLs and object-storage expectations.
- Suggested approach: Add a short code comment/doc note that the function is reserved, or remove it in a dedicated cleanup if not planned.
- Acceptance criteria: Maintainers can tell whether the function is dead code or reserved future behavior.
- Risk: Low.
- Suggested labels: refactor, docs, resourcepacks, P3

## Suspicious Areas That Need Human Judgment

- `AWS_EC2_METADATA_DISABLED` defaults to `true`; this is safer for many containers but may surprise direct EC2 users expecting instance-profile credentials.
- `*_REMOVE_EXTRA=true` generic safety checks only require a non-empty S3 prefix; for mods/configs/datapacks/resourcepacks, non-authoritative placeholder objects may still allow destructive local pruning.
- Reset backup filenames use one-second timestamp precision; repeated resets in the same second could collide depending on execution path and filesystem timing.
- GPU runtime installs AWS CLI v2 manually while CPU runtimes use Debian `awscli`; behavior/version skew may matter for S3-compatible endpoints.
