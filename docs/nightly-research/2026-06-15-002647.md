# Nightly Triage Memo

## Summary

- World reset backups are unsafe for custom `LEVEL_NAME`: reset targets the custom world directory, but backup creation archives literal `world`.
- S3 datapack sync can be overwritten by later `/datapacks` activation, so precedence is ambiguous.
- PR CI only builds `runtime-jre21`; recent GPU AWS CLI packaging changes are not caught before merge.
- Some examples still reference `:latest`, but publish workflow emits runtime target tags such as `runtime-jre21`.
- Several MinIO `mc` remediation docs are stale after the AWS CLI migration.
- Plugin S3 sync has duplicated temp-file cleanup code that likely leaks one temp file per plugin sync.

## High Priority Issue Candidates

### 1. Custom `LEVEL_NAME` world reset backup archives the wrong directory

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Operators using `LEVEL_NAME` can delete `/data/<LEVEL_NAME>` while the backup captures `/data/world`, potentially leaving no backup of the deleted world.
- Evidence: `minecraft_world_dir` respects `LEVEL_NAME` / `LEVEL`, and `reset_world` deletes that derived directory. `create_world_reset_backup` runs `tar -czf ... -C "${data_dir}" world`. Existing custom world smoke sets `LEVEL_NAME=custom-world` only with `RESET_WORLD_BACKUP=false`.
- Why this matters: World reset is destructive, and the project philosophy treats world data and mounted volumes as user-owned persistent state.
- Suggested approach: Archive the actual selected world basename from `minecraft_world_name`, validate it remains under `DATA_DIR`, and add custom-level backup smoke coverage.
- Acceptance criteria: With `LEVEL_NAME=custom-world` and backup enabled, reset produces an archive containing `custom-world/level.dat`; it does not archive unrelated `world/`; backup failure still prevents deletion.
- Risk: Medium; this touches destructive reset behavior.
- Suggested labels: bug, world-safety, backup, P1

### 2. S3 datapack sync can be replaced by local `/datapacks` activation

- Type: bug
- Priority: P1
- Confidence: medium
- Impact: `DATAPACKS_S3_BUCKET` syncs into `${DATA_DIR}/world/datapacks`, then `activate_datapacks` can atomically replace the same destination from `/datapacks` if that input directory is non-empty.
- Evidence: `install_phase.sh` calls `install_datapacks` then `activate_datapacks`. `install_datapacks` writes to `${DATA_DIR}/world/datapacks`; `activate_datapacks` calls `activate_dir "/datapacks" "${world_dir}/datapacks"`.
- Why this matters: GitOps users may expect S3 datapacks to be authoritative after sync, but mounted/input datapacks can silently win later in the same install phase.
- Suggested approach: Decide whether datapacks should follow staged asset semantics or direct-world S3 semantics, then make install and activation consistent.
- Acceptance criteria: A smoke test covers S3 datapack sync plus non-empty `/datapacks` and proves the intended precedence.
- Risk: Medium; changing precedence may affect existing deployments.
- Suggested labels: bug, s3, datapacks, lifecycle, P1

### 3. PR CI does not build the GPU runtime that recently changed AWS CLI packaging

- Type: infra
- Priority: P1
- Confidence: high
- Impact: `runtime-jre25-gpu` packaging regressions can merge before publish catches them, especially around AWS CLI availability for S3 sync.
- Evidence: Recent history includes GPU AWS CLI fixes, while `runtime-smoke.yml` builds only `runtime-jre21`. Full target matrix builds happen in `publish.yml` on main, tags, or manual dispatch.
- Why this matters: GPU runtime is advertised as a supported image and uses a different base/package path from CPU runtimes.
- Suggested approach: Add PR smoke coverage for `runtime-jre25-gpu`, or create a lightweight packaging check that verifies `aws --version`, `/entrypoint.sh`, and sourced libraries.
- Acceptance criteria: Pull requests fail before merge if `runtime-jre25-gpu` lacks AWS CLI or entrypoint dependencies.
- Risk: Low to medium; CI time may increase.
- Suggested labels: infra, ci, gpu, s3, P1

## Low Priority / Backlog Candidates

### 1. Plugin S3 sync likely leaks one temporary file per run

- Type: refactor
- Priority: P3
- Confidence: medium
- Impact: Plugin sync can leave temp files behind, usually small but avoidable.
- Evidence: `install_plugins` initializes `tmp_remote` and `tmp_remote_jars`, registers `cleanup_plugins_tmp`, then later redeclares temp variables and redefines the cleanup function to clean `tmp_remote` and `tmp_remote_topjars`. The first `tmp_remote_jars` file is no longer referenced by the final cleanup trap.
- Why this matters: The repo already has focused temp cleanup work for S3/world/reset paths, and plugin sync should follow the same standard.
- Suggested approach: Remove the dead first temp block and keep one cleanup helper/trap for the actual temp files used.
- Acceptance criteria: Plugin sync smoke proves temp files are cleaned on success and listing failure.
- Risk: Low.
- Suggested labels: refactor, plugins, cleanup, P3

- Suggested labels: refactor, plugins, P3
