# Nightly Triage Memo

## Summary

- Custom `LEVEL_NAME` resets delete the selected world but back up literal `/data/world`, creating a direct world-loss risk.
- Mods and config activation replace complete `/data` directories even when `*_REMOVE_EXTRA=false`.
- Plugin synchronization can delete a working staged plugin before its replacement downloads successfully.
- Datapack installation and activation ignore custom `LEVEL_NAME`.
- Image publication is not gated on successful CI for the exact commit or tag.
- AWS CLI behavior and runtime dependencies differ between CPU and GPU images without equivalent image-level coverage.

## High Priority Issue Candidates

### 1. Custom world reset backs up the wrong directory

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Resetting `LEVEL_NAME=custom-world` can delete `/data/custom-world` without backing it up.
- Evidence: `reset_world` derives deletion through `minecraft_world_dir`, while `create_world_reset_backup` runs `tar -czf ... -C "${data_dir}" world`. The custom-level test disables backups.
- Why this matters: The backup is the final protection before destructive deletion.
- Suggested approach: Pass the validated world basename to the backup helper and archive that exact directory.
- Acceptance criteria: A test with custom `LEVEL_NAME` and backups enabled verifies the archive contains the selected world, and deletion occurs only after successful backup.
- Risk: Medium; this changes destructive reset behavior.
- Suggested labels: bug, world-safety, backup, P1

### 2. Remove-extra false does not preserve local mods or configs

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Destination-only files under `/data/mods` or `/data/config` can disappear during startup.
- Evidence: `activate_dir` stages only source contents and atomically replaces the complete destination. `activate_mods` and `activate_configs` use it regardless of their remove-extra settings.
- Why this matters: Documentation presents `*_REMOVE_EXTRA=false` as the preservation-oriented default.
- Suggested approach: Merge source into the destination when remove-extra is false; reserve authoritative replacement for explicit remove-extra mode.
- Acceptance criteria: Destination-only files survive with remove-extra false and are removed only when remove-extra is true.
- Risk: High; some operators may depend on the current implicit replacement.
- Suggested labels: bug, persistence, s3, mods, configs, P1

### 3. Plugin download failures can remove working plugin files

- Type: bug
- Priority: P1
- Confidence: high
- Impact: A transient S3 failure can remove a valid staged plugin and prevent it from being activated.
- Evidence: `install_plugins` executes `safe_rm_f "${dest}"` before downloading the replacement. Non-strict mode can then continue successfully after download errors.
- Why this matters: A temporary object-storage failure should not destroy the last working plugin copy.
- Suggested approach: Download to a temporary file, validate success, then atomically replace the existing staged jar.
- Acceptance criteria: A failed replacement download leaves the original jar byte-for-byte intact in both strict and non-strict modes.
- Risk: Low to medium.
- Suggested labels: bug, plugins, s3, persistence, P1

### 4. Publish workflow is not gated by successful CI

- Type: release
- Priority: P1
- Confidence: high
- Impact: Main pushes can publish moving image tags while lint or runtime smoke is failing; tag pushes are not tested by the branch-only test workflows.
- Evidence: `publish.yml` triggers independently on main and `v*` pushes, with no dependency on the lint or runtime-smoke results.
- Why this matters: Broken images can replace production-facing tags.
- Suggested approach: Publish only after successful tests for the exact commit SHA.
- Acceptance criteria: Any failed required check prevents all image pushes, including versioned tag publication.
- Risk: Medium; workflow event and credential handling require careful validation.
- Suggested labels: release, ci, docker, reliability, P1

## Medium Priority Issue Candidates

### 1. Datapacks ignore custom world names

- Type: bug
- Priority: P2
- Confidence: high
- Impact: With `LEVEL_NAME=custom-world`, datapacks are written or activated under `/data/world/datapacks` instead of the active world.
- Evidence: `install_datapacks` and `activate_datapacks` hard-code `${DATA_DIR}/world`; world installation and reset use `minecraft_world_dir`.
- Why this matters: The server can start successfully while silently ignoring configured datapacks.
- Suggested approach: Resolve the datapack destination through `minecraft_world_dir`.
- Acceptance criteria: Default and custom-level tests verify datapacks reach the selected world's `datapacks` directory.
- Risk: Medium; existing misplaced files may require migration guidance.
- Suggested labels: bug, datapacks, world-paths, P2

### 2. Plugin partial failures can trigger destination removal during activation

- Type: bug
- Priority: P2
- Confidence: high
- Impact: With `PLUGINS_REMOVE_EXTRA=true`, activation can remove destination jars missing from an incomplete staging set.
- Evidence: Installation skips its own removal pass after errors, but non-strict mode returns success. `activate_plugins` independently removes destination jars absent from staging.
- Why this matters: Failure state is not carried across the install/activation boundary.
- Suggested approach: Publish staging only after successful synchronization or explicitly disable activation removal after partial failure.
- Acceptance criteria: A partial-download regression test proves existing destination plugins remain intact.
- Risk: Medium.
- Suggested labels: bug, plugins, s3, persistence, P2

### 3. Execute committed smoke tests currently omitted from CI

- Type: test
- Priority: P2
- Confidence: high
- Impact: Several existing regression tests receive syntax checking but no behavioral execution.
- Evidence: Workflows execute only part of `test/*.sh`; omitted examples include install-phase, marker validation, reinstall mismatch, server-properties mapping/detection, and Velocity ownership tests.
- Why this matters: These cover install ordering, configuration, ownership, and artifact safety.
- Suggested approach: Explicitly execute every active smoke test or maintain a documented exclusion list.
- Acceptance criteria: Each active test runs in CI and failures block merging.
- Risk: Low; dormant tests may initially reveal drift.
- Suggested labels: test, ci, P2

### 4. GPU image lacks an explicit tar dependency

- Type: infra
- Priority: P2
- Confidence: high
- Impact: World reset backups may fail on `runtime-jre25-gpu` if the CUDA base image does not happen to provide `tar`.
- Evidence: CPU `runtime-base` explicitly installs `tar`; the GPU package list does not. World reset backup invokes `tar`.
- Why this matters: Runtime behavior should not depend on undocumented transitive contents of the CUDA base image.
- Suggested approach: Explicitly install `tar` and check it in the GPU packaging smoke.
- Acceptance criteria: The GPU image asserts `tar` is available and exercises backup creation.
- Risk: Low.
- Suggested labels: infra, docker, gpu, backup, P2

### 5. Add image-level AWS CLI migration coverage

- Type: test
- Priority: P2
- Confidence: high
- Impact: Packaging and behavior differences between distro AWS CLI and AWS CLI v2 may escape mocked helper tests.
- Evidence: CPU images install Debian `awscli`; the GPU image installs AWS CLI v2. Current image checks primarily run `aws --version`.
- Why this matters: Endpoint handling, credentials, listing, sync, and delete conversion are core MinIO Client migration points.
- Suggested approach: Exercise the same mocked S3 argument matrix inside CPU and GPU images.
- Acceptance criteria: Both image families cover endpoint URLs, legacy credentials, copy, list, sync, and remove-extra conversion.
- Risk: Low.
- Suggested labels: test, docker, s3, migration, P2

### 6. Kubernetes examples inconsistently apply shutdown safeguards

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Users copying the hardcore and GPU manifests receive no RCON preStop hook, readiness probe, or explicit termination grace period.
- Evidence: The Paper and basic Fabric examples include these controls; the hardcore examples do not.
- Why this matters: These examples mount persistent world volumes and are presented as runnable configurations.
- Suggested approach: Apply the established lifecycle pattern consistently or label incomplete manifests clearly.
- Acceptance criteria: Persistent-world examples include graceful shutdown, readiness, sufficient grace time, and single-writer rollout guidance.
- Risk: Low.
- Suggested labels: docs, kubernetes, shutdown, P2

## Low Priority / Backlog Candidates

### 1. Document a safe backup restore procedure

- Type: docs
- Priority: P3
- Confidence: high
- Impact: Operators can create reset backups but lack an authoritative restore sequence.
- Evidence: Reset and world-install safety are extensively documented, but restoration is not.
- Why this matters: Improvised restoration can overwrite an active world or restore the wrong `LEVEL_NAME`.
- Suggested approach: Document archive inspection, stopping all writers, preserving current state, ownership checks, and controlled restart with fish-compatible commands.
- Acceptance criteria: Documentation covers default and custom world names without Bash-only heredocs.
- Risk: Low.
- Suggested labels: docs, backup, restore, operations, P3

### 2. Remove or document the ignored MAX_MEMORY example variable

- Type: docs
- Priority: P3
- Confidence: high
- Impact: Operators may assume `MAX_MEMORY` controls JVM heap size.
- Evidence: GPU examples set `MAX_MEMORY`, while runtime configuration uses `JVM_XMS` and `JVM_XMX`.
- Why this matters: Incorrect heap assumptions can cause Kubernetes OOM termination.
- Suggested approach: Remove it or implement a documented compatibility alias.
- Acceptance criteria: Examples contain only effective memory variables.
- Risk: Low.
- Suggested labels: docs, kubernetes, jvm, P3

### 3. Deduplicate plugin synchronization scaffolding

- Type: refactor
- Priority: P3
- Confidence: high
- Impact: Duplicate temporary variables, cleanup functions, traps, and Spigot conditions make destructive synchronization logic harder to audit.
- Evidence: `install_plugins` defines cleanup state twice and repeats type conditions.
- Why this matters: Complexity increases regression risk in plugin replacement and deletion.
- Suggested approach: Perform a mechanical cleanup after adding failure-path tests.
- Acceptance criteria: No behavior or log changes; plugin tests remain green.
- Risk: Medium without stronger tests.
- Suggested labels: refactor, plugins, P3

## Suspicious Areas That Need Human Judgment

- World installation replaces a nonempty selected world directory when it lacks `level.dat`; partial or damaged worlds may therefore be deleted without a reset flag.
- `AWS_EC2_METADATA_DISABLED` defaults to true, potentially disabling EC2 instance-profile credentials.
- The documented 120-second Kubernetes grace period may be insufficient when all RCON commands exhaust retries.
- `activate_resourcepacks` exists but is intentionally absent from the install flow.
- Resourcepack URL generation validates URL shape but not remote object existence.
- World archive discovery accepts exactly one direct `.zip` beneath a prefix; this is conservative but limits versioned archive layouts.
- Reset backup names have only one-second timestamp precision and use forced replacement when publishing the archive.

- Should moving image tags publish after every successful main build or only through explicit releases?
- Must EC2 instance-role credential discovery remain supported?
