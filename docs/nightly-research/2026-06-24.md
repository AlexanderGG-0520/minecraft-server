# Nightly Triage Memo

## Summary

- Custom `LEVEL_NAME` reset still appears to back up literal `/data/world` while deleting the configured world directory.
- `*_REMOVE_EXTRA=false` may not preserve destination-only mods/configs because activation replaces whole directories.
- Plugin S3 sync can remove an existing staged jar before a replacement download succeeds.
- Datapacks still target `/data/world` instead of the configured Minecraft world directory.
- AWS CLI migration coverage is mostly mocked and does not exercise real S3-compatible endpoint behavior.
- Several MinIO Client remediation docs still describe obsolete `mc` implementation details as current behavior.

## High Priority Issue Candidates

### 1. Custom world reset backs up the wrong directory

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Operators using `LEVEL_NAME` can delete `/data/<LEVEL_NAME>` while the backup captures `/data/world`.
- Evidence: `reset_world` deletes the directory returned by `minecraft_world_dir`, but `create_world_reset_backup` archives `world` with `tar -C "${data_dir}" world`. The custom-name reset smoke uses `RESET_WORLD_BACKUP=false`.
- Why this matters: This is a direct persistent-world safety issue in the destructive reset path.
- Suggested approach: Back up the validated selected world directory, not a hard-coded `world` path.
- Acceptance criteria: With `LEVEL_NAME=custom-world` and backup enabled, the archive contains `custom-world/level.dat`, excludes unrelated `/data/world`, and deletion happens only after backup success.
- Risk: Medium; reset behavior is destructive and must be tested narrowly.
- Suggested labels: bug, world-safety, backup, P1

### 2. Remove-extra false does not reliably preserve local mods/configs

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Destination-only files under `/data/mods` or `/data/config` can disappear during startup even when remove-extra is false.
- Evidence: `activate_dir` stages source contents and atomically replaces the complete destination. `activate_mods` and `activate_configs` use it independently of `MODS_REMOVE_EXTRA` / `CONFIGS_REMOVE_EXTRA`.
- Why this matters: The README frames remove-extra false as the safer default, but activation can still behave authoritatively.
- Suggested approach: Merge into destination when remove-extra is false; reserve full replacement for explicit remove-extra behavior.
- Acceptance criteria: Destination-only files survive with remove-extra false and are removed only when remove-extra true.
- Risk: High; current behavior may have accidental users.
- Suggested labels: bug, persistence, s3, mods, configs, P1

### 3. Plugin download failures can remove working staged jars

- Type: bug
- Priority: P1
- Confidence: high
- Impact: A transient object-storage failure can delete the previous staged plugin jar before activation.
- Evidence: `install_plugins` calls `safe_rm_f "${dest}"` before `s3_cp`. Non-strict mode can continue after errors.
- Why this matters: S3 failures should not destroy the last known-good plugin copy.
- Suggested approach: Download to a temporary file, verify success, then atomically replace the staged jar.
- Acceptance criteria: A failed replacement download leaves the prior jar byte-for-byte intact in strict and non-strict modes.
- Risk: Low to medium.
- Suggested labels: bug, plugins, s3, persistence, P1

## Medium Priority Issue Candidates

### 1. Datapacks ignore custom world names

- Type: bug
- Priority: P2
- Confidence: high
- Impact: With `LEVEL_NAME=custom-world`, datapacks can be written to `/data/world/datapacks` while the server uses `/data/custom-world`.
- Evidence: `install_datapacks` and `activate_datapacks` hard-code `${DATA_DIR}/world`; world install/reset use `minecraft_world_dir`.
- Why this matters: Startup can look successful while datapacks are silently ignored.
- Suggested approach: Resolve datapack paths through `minecraft_world_dir`.
- Acceptance criteria: Default and custom-world tests verify datapacks land in the active world directory.
- Risk: Medium; existing misplaced datapacks may need migration notes.
- Suggested labels: bug, datapacks, world-paths, P2

### 2. Add real S3-compatible AWS CLI migration coverage

- Type: test
- Priority: P2
- Confidence: high
- Impact: Mocked `aws` tests may miss endpoint, credential, sync, delete, pagination, or provider-specific behavior.
- Evidence: Current S3 tests mock `aws`; README advertises MinIO, R2, Garage, B2, Wasabi, and AWS S3. Recent history replaced MinIO Client with AWS CLI.
- Why this matters: Object storage is a core feature and migration risk area.
- Suggested approach: Add optional manual or scheduled tests against disposable MinIO or another controlled S3-compatible endpoint.
- Acceptance criteria: A non-default workflow validates list, copy, sync, endpoint URL, credential mapping, and remove-extra safety without production credentials.
- Risk: Medium; integration tests need careful secret and cleanup boundaries.
- Suggested labels: test, s3, migration, aws-cli, P2

### 3. CPU and GPU images use different AWS CLI sources

- Type: infra
- Priority: P2
- Confidence: high
- Impact: S3 behavior can differ between CPU images using Debian `awscli` and GPU image using AWS CLI v2.
- Evidence: `runtime-base` installs `awscli` from apt; `runtime-jre25-gpu` downloads and installs AWS CLI v2. CI mostly checks `aws --version` and mocked helper behavior.
- Why this matters: Provider compatibility should not depend on runtime flavor.
- Suggested approach: Run the same image-level S3 argument and sync smoke matrix in both CPU and GPU targets.
- Acceptance criteria: Both image families cover endpoint URL, legacy variables, region defaults, cp, list, sync, and remove-extra conversion.
- Risk: Low.
- Suggested labels: test, docker, gpu, s3, P2

### 4. Publish workflow is not gated on successful test workflows

- Type: release
- Priority: P2
- Confidence: high
- Impact: A main or tag push can publish images even if lint or runtime smoke would fail for that commit.
- Evidence: `publish.yml` runs independently on `main`, `v*`, and manual dispatch; lint/runtime-smoke are separate workflows.
- Why this matters: Moving image tags are production-facing.
- Suggested approach: Gate publication on required checks for the exact commit SHA, or move publishing behind a workflow-run release gate.
- Acceptance criteria: Failed lint/runtime smoke prevents all image pushes, including versioned tags.
- Risk: Medium; release workflow event wiring is easy to get subtly wrong.
- Suggested labels: release, ci, docker, reliability, P2

## Low Priority / Backlog Candidates

### 1. Mark obsolete MinIO Client docs as historical

- Type: docs
- Priority: P3
- Confidence: high
- Impact: Maintainers may follow outdated guidance about `mc-builder`, `/usr/local/bin/mc`, `configure_mc_alias`, and `mc mirror`.
- Evidence: Current code uses AWS CLI helpers, but several `docs/minio-mc-*` and older boundary docs still describe MinIO Client as current.
- Why this matters: The project now explicitly prefers AWS CLI-compatible S3 behavior.
- Suggested approach: Add archival banners or rewrite those docs as historical migration records.
- Acceptance criteria: Searches for old MinIO Client terms clearly distinguish historical notes from current implementation.
- Risk: Low.
- Suggested labels: docs, s3, migration, P3

### 2. Execute all committed smoke tests or document exclusions

- Type: test
- Priority: P3
- Confidence: medium
- Impact: Some committed smoke tests receive syntax checks but do not appear to run as behavioral CI steps.
- Evidence: Existing tests include install-phase, marker validation, reinstall mismatch, server-properties env mapping/detection, and Velocity ownership smoke files; workflow execution is selective.
- Why this matters: Dormant tests can drift and give false confidence.
- Suggested approach: Run every active `test/*.sh` explicitly or maintain a documented exclusion list.
- Acceptance criteria: Every active smoke test either runs in CI or has a documented reason for exclusion.
- Risk: Low; enabling dormant tests may reveal existing drift.
- Suggested labels: test, ci, P3

### 3. Remove or implement ignored `MAX_MEMORY` examples

- Type: docs
- Priority: P3
- Confidence: high
- Impact: Operators may think `MAX_MEMORY` controls heap sizing.
- Evidence: Examples set `MAX_MEMORY`, while JVM heap settings are driven by `JVM_XMS` and `JVM_XMX`.
- Why this matters: Misunderstood heap limits can lead to poor scheduling or OOM behavior.
- Suggested approach: Replace `MAX_MEMORY` with documented JVM variables, or implement it as an intentional compatibility alias.
- Acceptance criteria: Examples contain only effective memory settings, or docs explain the alias.
- Risk: Low.
- Suggested labels: docs, jvm, examples, P3

## Suspicious Areas That Need Human Judgment

- `activate_resourcepacks` exists and is smoke-declared but is intentionally absent from the install flow; this may be reserved future behavior or dead code.
- World install can replace a nonempty selected world directory when `level.dat` is absent and S3 world install is configured.
- `AWS_EC2_METADATA_DISABLED` defaults to true, which may intentionally block instance-profile credentials.
- The common 120-second Kubernetes grace recommendation can be tight once preStop, RCON retries, save waits, and PID 1 fallback are all counted.
- Plugin sync has duplicated temp/trap scaffolding, increasing audit risk around deletion behavior.
- Reset backup archive names use one-second timestamp precision.
