# Nightly Triage Memo

## Summary

- Custom `LEVEL_NAME` resets can back up `/data/world` while deleting a different world directory.
- Plugin S3 updates delete existing JARs before replacement downloads succeed.
- Datapack installation ignores custom world names and always targets `/data/world/datapacks`.
- Several committed smoke tests covering high-risk behavior are not executed by CI.
- Custom mods/config staging directories are used for sync but ignored during activation.
- AWS CLI migration documentation and real S3-compatible integration coverage remain incomplete.

## High Priority Issue Candidates

### 1. World reset backs up the wrong directory with custom `LEVEL_NAME`

- Type: bug
- Priority: P1
- Confidence: high
- Impact: A reset can delete `/data/<custom-world>` while preserving an unrelated `/data/world` backup.
- Evidence: `reset_world` derives `world_dir` through `minecraft_world_dir`, but `create_world_reset_backup` always runs `tar ... world`. Existing tests verify custom-world deletion only with backups disabled.
- Why this matters: The reset backup is the final recovery mechanism before destructive world deletion.
- Suggested approach: Archive the basename derived from the validated `world_dir`, preserving current staging and path validation.
- Acceptance criteria: With `LEVEL_NAME=custom-world`, the backup contains `custom-world/level.dat`; `/data/world` remains untouched; backup failure still prevents deletion.
- Risk: High because this changes destructive reset behavior.
- Suggested labels: bug, world-safety, backup, P1

### 2. Plugin sync removes working JARs before replacements download

- Type: bug
- Priority: P1
- Confidence: high
- Impact: A transient S3 failure can remove an existing plugin and leave no replacement.
- Evidence: `install_plugins` calls `safe_rm_f "${dest}"` before `s3_cp`. Non-strict mode then logs the failed download and continues.
- Why this matters: Best-effort synchronization should preserve the last known-good plugin during network failures.
- Suggested approach: Download into a temporary file beside the destination, validate successful completion, then atomically replace the existing JAR.
- Acceptance criteria: Failed downloads leave existing JAR contents unchanged; successful downloads replace them atomically; temporary files are cleaned.
- Risk: Medium; plugin update and remove-extra behavior must remain distinct.
- Suggested labels: bug, plugins, s3, persistence, P1

### 3. Datapack sync does not respect custom world names

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Datapacks may be installed into an unused or stale `/data/world` instead of the active custom world.
- Evidence: Both `install_datapacks` and `activate_datapacks` hard-code `${DATA_DIR}/world`; world install and reset use `minecraft_world_dir` and respect `LEVEL_NAME`/`LEVEL`.
- Why this matters: Operators can believe a datapack is active while the server loads another world directory.
- Suggested approach: Resolve the target through `minecraft_world_dir` consistently.
- Acceptance criteria: Custom and default world names receive datapacks in their actual `datapacks` directory, with tests ensuring unrelated worlds remain untouched.
- Risk: Medium because existing installations may contain files under the old hard-coded path.
- Suggested labels: bug, datapacks, world-safety, P1

## Medium Priority Issue Candidates

- Type: test
- Priority: P2

- Suggested approach: Execute every active smoke script explicitly or mark intentionally inactive scripts clearly.
- Acceptance criteria: CI runs all supported smoke scripts and reports failures by functional area.
- Risk: Low, although dormant tests may expose existing drift.
- Suggested labels: test, ci, regression, P2

### 2. Custom mods and config staging directories are ignored during activation

- Type: bug
- Priority: P2
- Confidence: high
- Impact: Assets can sync successfully into a custom directory but never reach `/data`, while stale default-directory contents are activated.
- Evidence: Installation uses `INPUT_MODS_DIR` and `INPUT_CONFIG_DIR`; activation reads hard-coded `/mods` and `/config`.
- Why this matters: The variables currently imply end-to-end staging-path support.
- Suggested approach: Use the same resolved path for sync and activation, with filesystem safety validation.
- Acceptance criteria: Custom paths work end-to-end and default paths retain current behavior.
- Risk: Medium due to existing mount conventions.
- Suggested labels: bug, assets, persistence, P2

### 3. Document a safe world-backup restore procedure

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Operators have an automated reset backup but no clear, safe recovery workflow.
- Evidence: Reset and world-install behavior are extensively documented, but no dedicated restore checklist exists.
- Why this matters: Backups are useful only when operators can restore them without colliding with a running pod or overwriting the wrong PVC path.
- Suggested approach: Document fish-compatible steps covering workload shutdown, archive inspection, world-name verification, extraction, permissions, and controlled restart.
- Acceptance criteria: Documentation includes default and custom world names and warns against restoring while a server writes to the volume.
- Risk: Low.
- Suggested labels: docs, backup, restore, kubernetes, P2

### 4. Add S3-compatible integration coverage for the AWS CLI migration

- Type: test
- Priority: P2
- Confidence: high
- Impact: Mocked tests may miss endpoint, pagination, path-style, delete, and AWS CLI version differences.
- Evidence: Current tests mock `aws`; CPU images use Debian `awscli`, while the GPU image installs AWS CLI v2 directly.
- Why this matters: S3 operations affect plugins, mods, configs, datapacks, resourcepacks, and world restoration.
- Suggested approach: Add a scheduled or manual disposable-endpoint workflow testing both CPU and GPU image paths.
- Acceptance criteria: Coverage includes legacy credential mapping, custom endpoints, nested prefixes, pagination, sync deletion, object download, and empty-source refusal.
- Risk: Medium because integration infrastructure and credentials require careful isolation.
- Suggested labels: test, s3, aws-cli, migration, P2

### 5. Reconcile shutdown grace guidance with worst-case RCON timing

- Type: docs
- Priority: P2
- Confidence: medium
- Impact: Kubernetes may terminate a pod before RCON retries and process-exit waits complete.
- Evidence: Examples commonly use 120 seconds, while defaults permit multiple timed RCON attempts plus `SHUTDOWN_WAIT_TIMEOUT=90` and `SHUTDOWN_TERM_WAIT=10`.
- Why this matters: Kubernetes includes `preStop` execution within the overall termination grace period.
- Suggested approach: Document the timeout budget and recommend a larger value for modded or large-world servers.
- Acceptance criteria: Shutdown documentation explains how RCON retry settings contribute to the required grace period.
- Risk: Low.
- Suggested labels: docs, kubernetes, shutdown, P2

## Low Priority / Backlog Candidates

### 1. Clean up duplicated plugin synchronization scaffolding

- Type: refactor
- Priority: P3
- Confidence: high
- Impact: Duplicate type checks, temporary variables, cleanup functions, and traps make the high-risk plugin path harder to audit.
- Evidence: `install_plugins` repeats Spigot conditions and defines temporary-file cleanup twice; the first allocated temporary file is no longer tracked by the second trap.
- Why this matters: Fragile cleanup code increases the likelihood of mistakes during future S3 changes.
- Suggested approach: Perform a mechanical cleanup after atomic plugin replacement is covered.
- Acceptance criteria: No temporary files remain after success or failure and plugin selection behavior is unchanged.
- Risk: Medium if mixed with functional changes.
- Suggested labels: refactor, plugins, cleanup, P3

### 2. Prevent same-second world backup filename collisions

- Type: design
- Priority: P3
- Confidence: medium
- Impact: Multiple resets within one second can publish the same backup filename using forced replacement.
- Evidence: Backup names have second-level timestamps and publication uses `safe_mv_f`, which maps to `mv -f`.
- Why this matters: Existing backups should not be silently overwritten.
- Suggested approach: Use collision refusal or a unique suffix while retaining recognizable timestamps.
- Acceptance criteria: An existing backup archive is never overwritten silently.
- Risk: Low.
- Suggested labels: backup, world-safety, design, P3

### 3. Mark obsolete MinIO Client documents as historical

- Type: docs
- Priority: P3
- Confidence: high
- Impact: Maintainers may mistake obsolete `mc-builder`, `mc alias`, and `mc mirror` instructions for current behavior.
- Evidence: Runtime code now uses AWS CLI, while several `docs/minio-mc-*` files still describe the removed client implementation.
- Why this matters: Repository documentation should distinguish historical remediation records from active guidance.
- Suggested approach: Add archival notices and link to `docs/s3-client-cleanup-boundary.md`.
- Acceptance criteria: Searches for MinIO Client instructions clearly identify them as historical.
- Risk: Low.
- Suggested labels: docs, minio, aws-cli, migration, P3

## Suspicious Areas That Need Human Judgment

- World archives are passed directly to `unzip` without an explicit entry or symlink safety preflight. The practical protection provided by the packaged `unzip` implementation should be verified.
- `rcon-stop` intentionally returns success to Kubernetes even when RCON fails. This preserves `preStop` compatibility but hides failure from Kubernetes event status.
- CPU and GPU images use different AWS CLI distribution channels and potentially different major versions.
- `activate_resourcepacks` exists but is intentionally not called by the install phase according to README behavior.
- World installation accepts exactly one direct-child ZIP under the configured prefix; operators may expect versioned archives or latest-file selection.
