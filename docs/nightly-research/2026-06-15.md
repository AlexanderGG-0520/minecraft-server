# Nightly Triage Memo

## Summary

- World reset backups appear unsafe for custom `LEVEL_NAME`: reset targets the custom world directory, but backup creation archives literal `world`.
- S3 datapack sync may be overwritten by the later `/datapacks` activation step.
- PR CI builds only `runtime-jre21`; recent GPU AWS CLI fixes are only fully exercised by publish after merge.
- Several MinIO `mc` remediation docs are now stale after the AWS CLI migration.
- The `paper-minio-assets` Kubernetes example uses `:latest`, but publish workflow appears to tag runtime targets, not `latest`.
- Restore/backup operation is under-documented compared with reset/install safety.

## High Priority Issue Candidates

### 1. Custom `LEVEL_NAME` world reset backup archives the wrong directory

- Type: bug
- Priority: P1
- Confidence: high
- Impact: Operators using `LEVEL_NAME` can get a backup of `/data/world` while the reset deletes `/data/<LEVEL_NAME>`, leaving no backup of the deleted world.
- Evidence: `reset_world` derives `world_dir` through `minecraft_world_dir`, but `create_world_reset_backup` runs `tar -czf ... -C "${data_dir}" world`. Existing smoke covers custom `LEVEL_NAME` only with `RESET_WORLD_BACKUP=false`.
- Why this matters: World reset is destructive. The project philosophy says world data and mounted volumes are user-owned persistent state.
- Suggested approach: Archive the actual selected world basename from `minecraft_world_name`, validate it is still inside `DATA_DIR`, and add a custom-level backup smoke.
- Acceptance criteria: With `LEVEL_NAME=custom-world` and backups enabled, reset produces an archive containing `custom-world/level.dat`; it does not archive unrelated `world/`; backup failure still prevents deletion.
- Risk: Medium; backup path changes touch destructive reset behavior.
- Suggested labels: bug, world-safety, backup, P1

### 2. S3 datapack sync can be replaced by local `/datapacks` activation

- Type: bug
- Priority: P1
- Confidence: medium
- Impact: `DATAPACKS_S3_BUCKET` syncs into `${DATA_DIR}/world/datapacks`, then `activate_datapacks` activates `/datapacks` into the same destination. If `/datapacks` is non-empty, it can replace freshly synced S3 datapacks.
- Evidence: `install_phase.sh` calls `install_datapacks` then `activate_datapacks`. `install_datapacks` writes to `${DATA_DIR}/world/datapacks`; `activate_datapacks` calls `activate_dir "/datapacks" "${world_dir}/datapacks"`.
- Why this matters: It can surprise GitOps users who expect S3 to be authoritative for datapacks.
- Suggested approach: Decide whether datapacks should follow mods/config staging semantics or direct-world semantics, then make install and activation consistent.
- Acceptance criteria: A smoke test covers S3 datapack sync plus non-empty `/datapacks` and proves the intended precedence.
- Risk: Medium; changing precedence may affect existing deployments.
- Suggested labels: bug, s3, datapacks, lifecycle, P1

### 3. PR CI does not build the GPU runtime that recently changed AWS CLI packaging

- Type: infra
- Priority: P1
- Confidence: high
- Impact: `runtime-jre25-gpu` packaging regressions can merge before being caught, especially around AWS CLI availability for S3 sync.
- Evidence: Recent history includes GPU AWS CLI fixes/removal commits `#200`, `#201`, and `#204`. `runtime-smoke.yml` builds only `runtime-jre21`; full matrix build/push happens in `publish.yml` on main/tags/workflow_dispatch.
- Why this matters: GPU runtime is a first-class advertised image and uses a different base/package path from CPU runtimes.
- Suggested approach: Add a PR smoke that builds `runtime-jre25-gpu` or a lightweight GPU packaging target and checks `aws --version`, `/entrypoint.sh`, and sourced libraries.
- Acceptance criteria: Pull requests fail before merge if `runtime-jre25-gpu` lacks AWS CLI or entrypoint dependencies.
- Risk: Low to medium; CI time may increase.
- Suggested labels: infra, ci, gpu, s3, P1

## Medium Priority Issue Candidates

### 1. MinIO `mc` remediation docs are stale after AWS CLI migration

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Maintainers may follow obsolete remediation guidance mentioning `mc-builder`, `MC_RELEASE`, `/usr/local/bin/mc`, and `configure_mc_alias`.
- Evidence: Dockerfile now installs AWS CLI and no longer has an `mc-builder`, while docs such as `docs/minio-mc-vulnerability-remediation-plan.md`, `docs/minio-mc-remediation-boundary.md`, and `docs/world-install-cleanup-boundary.md` still describe MinIO `mc` as current.
- Why this matters: Security and S3 migration work can be mis-scoped.
- Suggested approach: Add an archival note or update these docs to say they describe pre-AWS-CLI behavior.
- Acceptance criteria: Searching docs for `mc-builder`, `MC_RELEASE`, and `configure_mc_alias` no longer presents obsolete behavior as current.
- Risk: Low.
- Suggested labels: docs, s3, migration, P2

### 2. `paper-minio-assets` example appears to reference an unpublished `latest` image tag

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Users applying the example may pull a missing or unintended image.
- Evidence: `examples/kubernetes/paper-minio-assets/deployment.yaml` uses `ghcr.io/alexandergg-0520/minecraft-server:latest`; `publish.yml` tags matrix targets such as `runtime-jre21` and versioned target tags, not `latest`.
- Why this matters: This is a primary S3-compatible Kubernetes example.
- Suggested approach: Use an explicit runtime tag, likely `runtime-jre21`, matching other examples.
- Acceptance criteria: All examples reference tags that the publish workflow actually emits.
- Risk: Low.
- Suggested labels: docs, examples, release, P2

### 3. Backup exists, but restore workflow is not documented

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Operators can trigger reset backups but lack a clear, safe restore procedure.
- Evidence: README and docs describe reset backups and world install, but there is no dedicated restore procedure or validation checklist.
- Why this matters: Backup safety is incomplete without a tested recovery path.
- Suggested approach: Document manual restore steps for fish-compatible shells, including stopping the pod, preserving current data, extracting the chosen archive, and validating ownership.
- Acceptance criteria: Docs include a restore checklist and explicitly warn against restoring into a live server.
- Risk: Low.
- Suggested labels: docs, backup, restore, ops, P2

### 4. AWS CLI migration lacks real S3-compatible behavior coverage

- Type: test
- Priority: P2
- Confidence: medium
- Impact: Mocked tests may miss endpoint, region, path-style, pagination, and delete behavior differences across MinIO/R2/B2/Garage/Wasabi.
- Evidence: Current S3 smokes mock `aws`; README advertises multiple S3-compatible providers.
- Why this matters: The MinIO Client to AWS CLI replacement is a migration risk area.
- Suggested approach: Add an optional non-default integration workflow against disposable MinIO, covering cp/sync/list/remove-extra safety.
- Acceptance criteria: A manual or scheduled workflow validates AWS CLI behavior against MinIO without production credentials.
- Risk: Medium; integration tests can be flaky if over-scoped.
- Suggested labels: test, s3, migration, P2

- Suggested labels: design, docs, resourcepacks, P3
