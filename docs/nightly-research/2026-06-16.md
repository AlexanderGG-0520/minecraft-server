# Nightly Triage Memo

## Summary

- PR/runtime CI still builds only `runtime-jre21`, while recent history includes multiple GPU AWS CLI packaging fixes for `runtime-jre25-gpu`.
- Two Kubernetes Paper examples still use `ghcr.io/alexandergg-0520/minecraft-server:latest`, but the publish workflow appears to emit target-specific tags such as `runtime-jre21`, not `latest`.
- Several MinIO `mc` remediation docs are stale after the AWS CLI migration and still describe an `mc-builder` Dockerfile stage that no longer exists.
- `install_plugins` appears to allocate temporary files twice and loses the first `tmp_remote_jars` path, creating a likely temp-file leak.
- AWS CLI S3 behavior has good mocked smoke coverage, but no real S3-compatible integration coverage for MinIO/R2/Garage/path-style/pagination behavior.
- Resourcepack local sync behavior is intentionally non-activating per README, but `activate_resourcepacks` still exists and is source-smoked, which can confuse future maintainers.

## High Priority Issue Candidates

### 1. Add PR coverage for `runtime-jre25-gpu` AWS CLI availability

- Type: infra
- Priority: P1
- Confidence: high
- Impact: GPU runtime packaging regressions can merge and only fail during publish, even though the GPU image is advertised and recently changed.
- Evidence: `.github/workflows/runtime-smoke.yml` builds only `runtime-jre21`; `.github/workflows/publish.yml` includes `runtime-jre25-gpu`; recent commits include `fix: install aws cli for runtime-jre25-gpu` and `fix: remove awscli apt package from gpu runtime`.
- Why this matters: S3 sync now depends on AWS CLI, and the GPU image has a separate installation path using the upstream AWS CLI zip rather than the shared runtime-base apt package.
- Suggested approach: Add a lightweight PR smoke job that builds `runtime-jre25-gpu` or at least verifies the Dockerfile stage can install `aws`, `jq`, `rsync`, `mcrcon`, Java, and copied shell libraries.
- Acceptance criteria: Pull requests fail before merge if `runtime-jre25-gpu` lacks a working `aws` binary or cannot source the entrypoint libraries.
- Risk: GPU image builds are slower and may increase CI cost; keep the check minimal if full runtime smoke is too expensive.
- Suggested labels: `ci`, `gpu`, `aws-cli`, `regression-risk`

### 2. Replace `:latest` in Kubernetes Paper examples with published runtime tags

- Type: docs
- Priority: P1
- Confidence: high
- Impact: Users following the Paper Kubernetes examples may deploy a tag that is not produced by the current publish workflow, or may get non-deterministic behavior if `latest` exists from older publishing behavior.
- Evidence: `examples/kubernetes/paper-pvc/deployment.yaml` and `examples/kubernetes/paper-minio-assets/deployment.yaml` use `ghcr.io/alexandergg-0520/minecraft-server:latest`; `.github/workflows/publish.yml` emits matrix target tags like `runtime-jre21` and versioned target tags.
- Why this matters: These are first-path Kubernetes examples for PVC, RCON shutdown, and S3-compatible assets. Example drift undermines GitOps predictability.
- Suggested approach: Update examples to use explicit runtime tags, probably `runtime-jre21` unless the example requires another Java version.
- Acceptance criteria: No runnable example uses `:latest` unless the publish workflow intentionally creates and documents that tag.
- Risk: Existing users may have copied the old example; release notes or changelog mention may be useful.
- Suggested labels: `docs`, `examples`, `kubernetes`, `image-tags`

## Medium Priority Issue Candidates

### 1. Archive or update stale MinIO `mc` remediation docs after AWS CLI migration

- Type: docs
- Priority: P2
- Confidence: high
- Impact: Maintainers may follow obsolete remediation guidance for `mc-builder`, `MC_RELEASE`, and `mc` command behavior even though runtime S3 now uses AWS CLI.
- Evidence: `Dockerfile` installs `awscli` in CPU runtime-base and AWS CLI v2 in the GPU stage; `scripts/lib/s3_client.sh` wraps `aws`; docs such as `docs/minio-mc-remediation-boundary.md` and `docs/minio-mc-vulnerability-remediation-plan.md` still describe `mc-builder`.
- Why this matters: The repo explicitly prefers AWS CLI-compatible S3 behavior and wants MinIO Client kept out unless justified.
- Suggested approach: Add clear archival headers to old `minio-mc-*` docs or replace them with AWS CLI migration/security notes.
- Acceptance criteria: Searching docs for `mc-builder`, `MC_RELEASE`, and `mc alias` makes it clear these are historical records, not current implementation guidance.
- Risk: Historical vulnerability records should not be deleted without preserving audit context.
- Suggested labels: `docs`, `maintenance`, `aws-cli`, `minio-migration`

### 2. Clean up duplicated plugin temp-file setup and add temp cleanup coverage

- Type: bug
- Priority: P2
- Confidence: medium
- Impact: Plugin S3 sync may leak at least one temporary file per run and the duplicated cleanup logic increases future maintenance risk.
- Evidence: `scripts/lib/content_assets.sh` initializes `tmp_remote` and `tmp_remote_jars`, creates both with `mktemp`, then later redeclares `tmp_remote` and introduces `tmp_remote_topjars`; the second cleanup trap no longer tracks `tmp_remote_jars`.
- Why this matters: Plugin sync is part of the S3 asset path and runs during startup. Leaks are bounded by container lifecycle but still indicate fragile cleanup logic.
- Suggested approach: Remove the unused first temp-file block or consolidate temp allocation into one block with one cleanup trap.
- Acceptance criteria: Plugin sync smoke verifies no `install_plugins` temp files remain after success, list failure, and download failure.
- Risk: Be careful not to change the intentional top-level `.jar` only plugin sync policy or Paper `.paper-remapped` protection.
- Suggested labels: `bug`, `s3`, `plugins`, `cleanup`

### 3. Add optional real S3-compatible integration coverage for AWS CLI migration

- Type: test
- Priority: P2
- Confidence: high
- Impact: Mocked AWS CLI tests may miss endpoint, region, path-style, pagination, delete semantics, and provider-specific behavior across MinIO, R2, Garage, B2, or Wasabi.
- Evidence: `test/s3-client-smoke.sh` and related workflow tests mock `aws`; README advertises multiple S3-compatible stores; recent history replaced MinIO Client with AWS CLI.
- Why this matters: The AWS CLI replacement is a migration risk in exactly the project’s high-risk asset and world install paths.
- Suggested approach: Add a manual or scheduled non-default integration workflow against a disposable S3-compatible endpoint.
- Acceptance criteria: The workflow validates list, cp, sync, endpoint URL, credentials mapping, remove-extra empty-source refusal, and at least one nested prefix case.
- Risk: Requires secrets or a disposable service; keep it opt-in and never run against production buckets.
- Suggested labels: `test`, `s3`, `aws-cli`, `integration`

## Low Priority / Backlog Candidates

### 1. Clarify local resourcepack activation ownership

- Type: docs
- Priority: P3
- Confidence: medium
- Impact: Maintainers may assume `activate_resourcepacks` is part of the install phase because it exists and is source-smoked, while README says resourcepacks are synced locally but not activated into `/data/resourcepacks`.
- Evidence: `scripts/lib/install_phase.sh` calls `install_resourcepacks` but not `activate_resourcepacks`; `scripts/lib/content_assets.sh` defines both; `test/install-phase-smoke.sh` expects only `install_resourcepacks`.
- Why this matters: Resource pack behavior affects client-facing URLs and server.properties safety.
- Suggested approach: Either document `activate_resourcepacks` as intentionally unused/reserved, or remove it in a dedicated cleanup if it is dead code.
- Acceptance criteria: Future readers can tell whether local activation is intentionally disabled or an incomplete feature.
- Risk: Removing the function may break external sourcing assumptions if anyone uses it manually.
- Suggested labels: `docs`, `resourcepacks`, `cleanup`

### 2. Decide whether CPU and GPU AWS CLI installation should be aligned

- Type: design
- Priority: P3
- Confidence: medium
- Impact: CPU images use distro `awscli`, while GPU installs AWS CLI v2 directly. Behavior/version drift could produce different S3 behavior across image flavors.
- Evidence: `Dockerfile` runtime-base installs `awscli`; `runtime-jre25-gpu` downloads `awscli-exe-linux-${aws_arch}.zip`.
- Why this matters: Operators should not see S3 sync differences only because they chose the GPU image.
- Suggested approach: Record whether this divergence is intentional. If not, standardize or add smoke assertions that print/verify the AWS CLI major version per target.
- Acceptance criteria: Maintainers can identify AWS CLI version expectations for each published runtime image.
- Risk: Aligning packaging may increase image size or dependency complexity.
- Suggested labels: `design`, `docker`, `aws-cli`, `compatibility`
